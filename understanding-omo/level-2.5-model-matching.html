第二层半 · 模型匹配深度专题

为什么每个 Agent 需要特定的模型，以及如何自定义而不破坏系统。

核心洞察：模型即开发者

把 AI 模型想象成开发团队里的成员。每个模型有不同的思维模式、不同的个性、不同的优势。模型不是简单的好坏之分，而是思考方式不同。给 Claude 和 GPT 同样的指令，它们的理解方式截然不同。

这不是 bug，而是整个系统的基础设计。Oh My OpenAgent 为每个 Agent 分配与其工作风格匹配的模型——就像组建团队时，把每个人放在最适合他们个性的岗位上。

Sisyphus：社交型技术主管

Sisyphus 是那种认识所有人、到处走动、通过沟通和协调把事情办成的开发者。他和其他 Agent 交谈、理解整个代码库的上下文、智能地委派工作，编码能力也不错。但纯粹的深层技术难题？他会有点吃力。

这就是为什么 Sisyphus 使用 Claude / Kimi / GLM。这些模型擅长：

  • 遵循复杂的多步骤指令（Sisyphus 的提示词约 1100 行）
  • 在大量工具调用中维持对话流
  • 理解细微的委派和编排模式
  • 生成结构良好、富有沟通性的输出

用 Sisyphus 配合较旧的 GPT 模型，就像把你最优秀的项目经理——那个协调所有人、主持站会、让整个团队保持一致的人——塞进一个房间里独自调试竞态条件。错配。GPT-5.4 和 GPT-5.5 现在有专用的 Sisyphus 提示路径，但 GPT 仍然不是编排器的推荐默认模型。

Hephaestus：深度专家

Hephaestus 是那种整天待在房间里编码的开发者。话不多，可能看起来有点社交障碍。但给他一个硬核技术难题，他会在三小时后带着一个别人找不到的解决方案出现。

这就是为什么 Hephaestus 使用 GPT-5.5。GPT-5.5 为此而生：

  • 深度自主探索，无需手把手指导
  • 跨复杂代码库的多文件推理
  • 原则驱动执行（给目标，不是给配方）
  • 长时间独立工作

用 Hephaestus 配合 GLM 或 Kimi，就像把你最善于沟通、社交活跃的开发者分配去独自做纯技术工作。他们最终也能完成，但不会发光——你浪费了让他们有价值的技能。

要点总结

每个 Agent 的提示词都针对其模型的个性精心调整。当你更换模型时，你更换了大脑——同样的指令会被完全不同的方式理解。模型匹配不是关于"更好"或"更差"，而是关于是否合适。

Claude 和 GPT 的思考差异

这很重要，因为它解释了为什么有些 Agent 同时支持两个模型族而有些不行。

Claude 对机制驱动的提示响应更好——详细的检查清单、模板、逐步流程。规则越多 = 依从性越好。你可以写一个 1100 行的提示词，包含嵌套工作流，Claude 会遵循每一步。

GPT（尤其是 5.2+）对原则驱动的提示响应更好——简洁的原则、XML 结构、明确的决策标准。规则越多 = 矛盾面越多 = 偏离越严重。GPT 在你陈述目标、让它自己找出机制时表现最好。

Prometheus 曾经为这种差异维护两套独立的提示。现在它使用单个薄提示配合 ulw-plan，因此切换模型只改变回退选择，不改变提示文件。Atlas 仍然支持按模型族的提示行为；Prometheus 不在运行时会自动切换提示。

已测试模型清单

🚨 阅读本文前请先看这里 — Sisyphus 不是"随便什么模型都能跑"的系统 🚨

SISYPHUS 仅在本文档列出的精确模型上经过测试和验证——其他模型一概未测试。支持的集合经过精心设计：

模型族已测试版本
Claude Fable 5 · Opus 4.8 · Opus 4.7 · Sonnet 4.6
Kimi K2.7 · K2.6 · K2.5
GLM 5 / 5.1（可接受——在长嵌套工作流上略松散）
GPT 5.4 / 5.5（有专用的 GPT 提示路径——受支持，但仍非编排器的推荐默认）

如果某个模型不在这个列表中，它在 Sisyphus 上是 100% 未测试、100% 未验证的。它可能根本不工作；也可能看起来工作，然后在三次工具调用后崩溃。如果你发现它凑合能用了——那是个奇迹。这不是受支持的配置，未经认可，也不能保证明天还能用。

对 Sisyphus 的每一次提示变更都是针对上述模型编写、调优和回归测试的——且仅针对这些模型。没有人关注未列表模型的表现。后果可不是微妙的：

  • 未列出的模型可能在下一个补丁时毫无征兆地崩溃。一个帮助了 Claude/Kimi 的提示调整，可能瞬间摧毁什么脆弱的东西——而我们永远不会注意到，因为我们没在测试它。别把它当成 bug 报告。它从来就不是故意能用的。
  • 提示无法修复模型。模型有硬性的、内在的特性。再多的提示雕琢也不能让一个模型做它本质上做不到的事。如果某个模型不适合编排，它就永远不适合——不管提示打磨得多完美。

强烈不建议在任何未列出的模型上运行 Sisyphus。如果你偏要这么做，完全后果自负——并且你应该预料到它会崩溃。

MiniMax / Qwen / MiMo / DeepSeek 作为 Sisyphus —— 千万别

我们没找到任何能让 MiniMax、Qwen、MiMo 或 DeepSeek 在 Sisyphus 角色上表现可接受的方法。我们试过了。它们在 Sisyphus 的嵌套 todo + 委派 + 编排提示下撑不住。这不是"再调调就行"的情况——上面说过了：提示无法修复模型。

MiniMax 和 Qwen 在 Sisyphus 角色上特别糟糕，我们几乎要完全禁止了。把"Sisyphus on MiniMax"和"Sisyphus on Qwen"当作你永远不应该碰的配置。（这些模型在其他地方仍有正当用途——MiniMax 用于快速工具回退，Qwen 用于视觉工作，下文都有说明——只是千万别作为编排器。）

查看可用模型

在配置任何内容之前，先看看当前系统能跑什么。

列出所有可用模型

opencode models

这会打印你当前可以使用的每个 provider/model 组合。提供商来源是你已连接的身份认证 + models.dev 目录。OpenCode 排序输出时 opencode* 提供商排在前面——这是有意为之，不是美观问题。

opencode auth list

显示你已经登录了哪些提供商。

推荐订阅方案

你不需要每个提供商。你只需要配好两个。

最优组合：OpenCode Go + OpenAI Plus/Pro

总计约 $30/月。在成本和覆盖范围上都优于直接订阅 Anthropic + OpenAI + Google（约 $60+/月）。

订阅费用获得内容覆盖范围
OpenCode Go $10/月 kimi-k2.5, kimi-k2.6, glm-5, glm-5.1, minimax-m2.5, minimax-m2.7, minimax-m3, mimo-v2-pro, qwen3.5-plus, qwen3.6-plus Claude 族替代（Kimi, GLM），Gemini 族替代（Qwen），工具/检索（MiniMax）
OpenAI Plus/Pro $20+/月 gpt-5.4, gpt-5.4-pro, gpt-5.5, gpt-5.5-codex GPT 原生 Agent（Hephaestus, Oracle, Momus），模型灵活 Agent 的 GPT 回退

为什么是这个组合

  1. Hephaestus 需要 GPT-5.5。它没有 Claude 族回退。ChatGPT Plus/Pro 或 OpenAI API 是最便宜的可行路径。
  2. OpenCode Go 覆盖了编排和创意场景。Kimi K2.5/2.6 在 Sisyphus/Atlas 上的表现类似 Claude。GLM-5 填补长尾需求。Qwen 在 Gemini 不可用时处理视觉任务。
  3. 没有哪个单一的提供商能覆盖一切。仅 Anthropic 的方案会导致 Hephaestus 不可用。仅 OpenAI 的方案会降低 Sisyphus 的表现。你至少需要每个族一个。

如果你已经有 Claude 订阅

在安装时添加 --claude=max20（或 yes）。Claude 链默认值（Opus 4.7，快照支持）会为 Sisyphus/Prometheus/Atlas 启用，你仍然免费获得 OpenCode Go 回退。可以固定到 claude-opus-4-8 或 claude-fable-5 来使用当前顶级 Claude 配合 Sisyphus/Atlas 调优提示；Prometheus 保持其单一 ulw-plan 支持的提示。最佳编排 + 预算安全网。

如果你没有任何订阅

仅 OpenCode Go 可以让 Sisyphus/Atlas/Oracle/Librarian/Explore 工作。没有 GPT 访问权，Hephaestus 不会激活，所以你会失去自主深度工作能力。建议尽快添加 ChatGPT Plus。

模型族回退链

当"原生"模型不可用时，oh-my-openagent 沿每个 Agent 的回退链逐个尝试直到连接成功。这些链硬编码在 packages/omo-opencode/src/shared/model-requirements.ts 中。不存在单一的全局优先级列表。每个 Agent 和每个 category 都有自己的链。

系统包含两个独立的回退机制：

  • model-fallback：主动解析，在 chat.params 中使用硬编码的 AGENT_MODEL_REQUIREMENTS 和 CATEGORY_MODEL_REQUIREMENTS 替换模型
  • runtime-fallback：被动恢复，响应 session.error 事件，可在 runtime-fallback hooks 中按 category/agent 配置

关于当前顶级模型与自动解析链的说明

两个东西以不同速度更新，这种差异解释了为什么下面仍能看到"Opus 4.7"作为默认值：

  • 当前的顶级模型是 Claude Fable 5 和 Opus 4.8，以及 Kimi K2.7。Sisyphus 和 Atlas 有针对这些模型专用的调优提示路径；Prometheus 保持一个 ulw-plan 支持的提示。你可以在配置中固定使用："anthropic/claude-opus-4-8"、"anthropic/claude-fable-5"、"opencode-go/kimi-k2.7"。想显式选择较新模型时使用。
  • 自动解析回退链下面的优先级仍以 Opus 4.7 和 Kimi K2.6 开头。这是有意为之，而不是过时的：链只自动选择捆绑能力快照所构建的模型，以确保变体和上下文窗口解析正确。它们在 Opus 4.8 / K2.7 进入模型目录后会提升为链默认值；在此之前，你可以通过显式命名来选用较新模型及其提示。

所以下面的链中的"Opus 4.7 (max)"条目是快照支持的基础，不是建议优先选择 4.7 而非 4.8。

Claude 族（沟通型，指令跟随）

使用者：Sisyphus, Atlas, Sisyphus-Junior, Metis（Claude 路径）, Prometheus（主要回退）, unspecified-low, unspecified-high。

配置风险说明

Sisyphus → MiniMax / Qwen 强烈不建议到"几乎禁止"的程度。两者在编排提示下都撑不住。永远不要把它们用作编排器。

Oracle → MiniMax 同样原因。Oracle 需要持续推理；MiniMax 会偏离。
Explore → Opus 巨大的成本浪费。Explore 需要速度，不是智能。

visual-engineering → Kimi/GLM 错误的推理风格。Gemini 不可用时用 Qwen，不要用 Claude 类模型。

模型解析流程

每个 Agent 都有一个回退链。系统按优先级顺序尝试模型，直到通过你连接的提供商找到一个可用的。你不需要为每个模型配置提供商。只需认证（opencode auth login），系统会计算出哪些模型可用以及在哪里。

解析管道（来自 packages/omo-opencode/src/shared/model-resolution-pipeline.ts）：

第 1 步：用户覆盖（Override）
用户显式配置或 UI 选择的模型（仅主 Agent）
第 2 步：Category 默认值
来自 category 配置（当 Agent 设置了 category 时）
第 3 步：用户 fallback_models
配置的字符串/对象，在硬编码链之前尝试
第 4 步：提供商回退
AGENT_MODEL_REQUIREMENTS / CATEGORY_MODEL_REQUIREMENTS
第 5 步：系统默认值
最终安全网
Agent 请求 → 用户覆盖（如有）→ 回退链 → 系统默认值

核心 Agent 的 tab 切换是确定性的，通过注入的运行时顺序字段实现。固定优先级顺序是 Sisyphus（order: 0）、Hephaestus（order: 1）、Prometheus（order: 2）、Atlas（order: 3），其余 Agent 随后排列。

你的显式配置总是优先。如果你为某个 Agent 设置了特定的模型，即使解析数据未缓存，该选择也优先。

Variant 和 reasoningEffort 覆盖会标准化为模型支持的值，因此跨提供商的覆盖会优雅降级而不是硬失败。

模型能力由 models.dev 支持，有可刷新的缓存和能力诊断。使用 bunx oh-my-opencode refresh-model-capabilities 更新缓存，或配置 model_capabilities.auto_refresh_on_start 在启动时刷新。

要查看你的 Agent 实际使用的模型，运行 bunx oh-my-opencode doctor。它基于你当前的身份认证和配置显示有效的模型解析结果。

基于文件的提示词

你可以使用 file:// URL 在 prompt 字段中加载 Agent 系统提示词，或使用 prompt_append 附加额外内容。prompt_append 字段也适用于 category。

{
  "agents": {
    "sisyphus": {
      "prompt": "file:///path/to/custom-prompt.md",
    },
    "oracle": {
      "prompt_append": "file:///path/to/additional-context.md",
    },
  },
  "categories": {
    "deep": {
      "prompt_append": "file:///path/to/deep-category-append.md",
    },
  },
}

文件内容在运行时加载并注入到 Agent 的系统提示词中。支持 ~ 扩展到家目录和相对 file:// 路径。

示例 D — 添加 DeepSeek 作为 GPT 替代

如果你有 OpenRouter 并希望在 GPT 不可用时将 DeepSeek 加入链：

{
  "agents": {
    "oracle": {
      "model": "openai/gpt-5.5",
      "variant": "high",
      "fallback_models": [
        "anthropic/claude-opus-4-8",
        { "model": "openrouter/deepseek/deepseek-v3.2", "temperature": 0.7 },
        "opencode-go/glm-5.1",
      ],
    },
  },
}

fallback_models 接受纯模型字符串和包含 variant、reasoningEffort、temperature、top_p、maxTokens、thinking 的逐个回退对象的混合数组。
