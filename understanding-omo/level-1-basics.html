OmO 是什么

Oh My OpenAgent（简称 OmO）是一个面向 AI 编程助手的多模型智能体编排插件。它不是一套提示词模板，也不是一个简单的工具集合——它是一个完整的智能体操作系统，运行在 OpenCode 之上，将单一的 AI 会话转变为一支协调的开发团队。

OmO 的核心理念很简单：不让任何一个模型做所有事。Claude Opus 擅长规划和编排，GPT-5.5 擅长深度推理和代码生成，Kimi K2.6 在中文理解和长上下文上表现优异，Gemini 在前端和视觉任务上有独特优势。OmO 让它们各司其职，自动协作，你不需要知道背后是谁在工作。

为什么需要 OmO

单一模型的锁死效应

如果你只用过一个 AI 编程助手，你可能觉得它足够好。但如果你用过两个以上，你会发现每个模型都有独特的优势和致命的弱点。Claude Code 在被采集长上下文时表现惊艳，但在改写已有代码时常出偏差。GPT 在做架构决策时思路开阔，但在严格执行既有模式上不够可靠。

问题在于：现有的工具都默认你只用一个模型。你的工作流、你的提示词、你的工具链——全绑定在一个供应商上。当更好的模型出现时，你被锁住了。当你需要的任务不适合当前模型时，你没有选择。

Anthropic 封锁事件

2025 年，Anthropic 封锁了 OpenCode，原因是 OmO 的存在让 Claude Code 暴露了竞争力上的短板。这个事件不是一个孤立的技术争端——它揭示了一个根本性的行业问题：AI 供应商希望你被锁在他们的墙内。

这不是阴谋论。这是商业逻辑。如果每个供应商都让你只能用自己的模型，他们就能控制定价、控制功能、控制你的迁移成本。

多模型编排是唯一出路

未来的 AI 编程不是选一个赢家。每个月都有新模型发布，价格越来越低，能力越来越强。没有哪个供应商能永远领先。OmO 的立场很明确：不做任何一家供应商的附属品。你的工作流属于你自己，OmO 只是那个让你自由选择、自由切换的编排层。

你不需要为 2 小时的活花 200 美元。未来的方向不是选一个赢家，而是编排所有赢家。

核心哲学

OmO 的架构和功能设计都建立在四条基本原则之上。这些原则不是口号——它们是每个功能取舍背后的决策标准。

人在回路 = 瓶颈

如果你需要不断修正 AI 的半成品代码、逐步指导它完成任务、反复澄清需求——那不是人机协作，那是 AI 未能履行其职责。就像自动驾驶需要人类接管方向盘是系统故障一样，OmO 的目标是让 agent 在无需人类干预的情况下完成工作。输入 ultrawork，然后去做别的事。

代码不可区分

Agent 写的代码应该与高级工程师写的代码无法区分。不是"AI 生成的代码需要清理"，不是"不错的起点"——而是真正的、最终的、可投入生产的代码。遵循已有模式、合理处理错误、不画蛇添足。OmO 的 Hashline 编辑工具和 Comment Checker 正是为此而设计。

最小化认知负担

人类只需要说他们想要什么。其余全是 agent 的工作。两种模式实现这一点：Prometheus 模式（访谈式规划，帮你理清真正想要什么）和 Ultrawork 模式（你说方向，它做剩下的一切）。人类提供意图，系统提供结构与执行。不多不少。

可预测 · 持续 · 可委派

理想的 agent 应该像编译器一样工作：Markdown 文档进去，工作代码出来。会话崩溃？/start-work 恢复。多天项目？上下文被保留。OmO 的 Todo Enforcer 确保 agent 不会中途闲置，Ralph Loop 确保任务不达 100% 不停止。像委派任务给一个可靠的队友一样委派给 agent。

两种工作模式

OmO 提供两种截然不同的工作模式，适应不同的任务类型和用户习惯。你可以把它们理解为"自动驾驶"和"对话式驾驶"——在不同的场景下各有优势。

Ultrawork 模式

只管做，别问

你输入 ultrawork 或 ulw，OmO 就进入"只管做"模式。Sisyphus 主编排器接管会话，激活所有可用 agent，构建 todo 列表，然后像一个永不疲倦的工程师一样从头做到尾。不需要你批准每一步，不需要你回答问题，不需要你中途介入。

适合：明确的任务、已知的技术方案、大规模的代码重构。

Prometheus 模式

先规划，再执行

输入 /start-work 调用 Prometheus。他不是直接开始写代码——而是像一位资深架构师一样采访你：你的目标是什么？约束条件是什么？有哪些潜在的陷阱？通过迭代提问理清需求后，他会输出一份结构化计划，经过 Metis 和 Momus 的审阅后，Sisyphus 再开始执行。

适合：复杂需求、不确定性高的任务、新项目的初始阶段。

IntentGate

传统 AI 编程工具有一个根本性的缺陷：它们把你说的话当作字面指令。你说"找到这个 bug"，它就去 grep 代码。你说"分析这段逻辑"，它就真的只分析不修改。但实际上，你说"找到这个 bug"的意思是"帮我修好它"，你说"分析这段逻辑"的意思是"看看有没有问题然后改掉"。

这就是字面理解的陷阱。人类在日常沟通中大量使用间接表达——我们期待对方理解我们的真实意图，而不是字面意思。

IntentGate 如何工作

IntentGate 是 OmO 的一条预处理管道，在用户输入到达任何 agent 之前先拦截分析。它不直接执行指令，而是先问一个问题：用户真正想要的是什么？

具体来说，IntentGate 将用户意图分类为以下四种之一：

类别特征典型输入
ultrawork执行类任务，需要端到端完成"把这个功能加上"、"重构这里"、"实现一个日志系统"
search信息查找，不需要修改代码"这个函数在哪里定义"、"帮我查一下 API"
analyze代码审查或逻辑分析"这段代码有什么问题"、"这个设计合理吗"
team需要多 agent 协作的复杂任务"并行调查这三个模块"、"做一个安全审计"

分类完成后，系统会根据类别选择最合适的处理路径——包括是否激活多 agent 并行、是否限制为只读操作、是否进入 Ultrawork 模式。这确保了同一个 input 不会被错误地送入错误的执行管道。

注意：IntentGate 是 Ultimate 版的特性。Light 版（Codex CLI）只识别 ultrawork/ulw 关键字，不做完整意图分类。

两版发行

OmO 不是一款单一产品。它面向两个不同的 AI 编程环境，分别打包为两个版本。它们的核心理念相同，但功能范围和使用方式不同。

Ultimate Edition

omo for OpenCode

OmO 的完整形态。11 个专业 agent、54-61 个生命周期 hook、20-39 个工具、5 个内置 MCP 服务器、完整的 Team Mode、Hashline 编辑、AST-Grep、后台 agent 并行、Claude Code 完全兼容——一切。

  • 11 Agent + 54-61 Hook
  • 5 内置 MCP + 20-39 工具
  • Team Mode（最多 8 并行成员）
  • Hashline 锚定编辑
  • Claude Code 完全兼容
  • Skill-Embedded MCP
bunx oh-my-openagent install

Light Edition

omo for Codex CLI (lazycodex)

OmO 在 Codex CLI 中的便携实现。包含 8 个核心组件（rules, comment-checker, git-bash, LSP, ultrawork, ulw-loop, start-work-continuation, telemetry）和插件级 MCP。没有 agent 编排层，没有 team_* 工具——Codex 自身的 shell 会话处理这些工作。

  • 8 个核心组件
  • rules, comment-checker, LSP
  • ultrawork, ulw-loop
  • 插件级 MCP 支持
  • 无 Agent 编排层
npx lazycodex-ai install

两者的关键区别不在于"谁更好"，而在于你用什么工具。如果你使用 OpenCode，Ultimate 版是你需要的一切。如果你使用 Codex CLI，Light 版提供了 OmO 核心体验的便携子集。你也可以通过 bunx oh-my-openagent install --platform=both 同时安装两个版本。

快速开始

安装 OmO 不需要复杂的配置流程。一行命令，然后一个词开始工作。

# Ultimate Edition（OpenCode 用户）
bunx oh-my-openagent install

# Light Edition（Codex CLI 用户）
npx lazycodex-ai install

安装完成后，在 OpenCode 或 Codex CLI 中输入以下内容开始：

# 一句话即可
ultrawork 帮我把这个项目的测试覆盖率达到 80% 以上

# 或简写
ulw 重构用户认证模块，改用 JWT
命令用途
bunx oh-my-openagent install安装 Ultimate 版（交互式向导）
npx lazycodex-ai install安装 Light 版（Codex CLI）
bunx oh-my-opencode doctor运行健康诊断（4 大类检查）
ultrawork / ulw进入端到端执行模式
/start-work启动 Prometheus 规划-执行流程
/ulw-loop启动持续执行循环
