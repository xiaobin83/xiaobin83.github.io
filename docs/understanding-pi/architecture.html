Pi 整体架构

项目概述

Pi 是一个极简、可扩展的终端编程助手（coding agent harness）。它以 CLI 工具 pi 为核心入口，通过交互式终端界面、print 模式、JSON 模式和 RPC 模式等多种方式与用户和外部系统交互。

Pi 的设计哲学是 "最小核心 + 可扩展"。核心只提供最基本的编程助手能力（读写文件、执行 bash、编辑文本），其他功能——子代理（sub-agents）、计划模式（plan mode）、权限弹窗、MCP 集成等——全部通过扩展（extensions）、技能（skills）、提示模板（prompt templates）和第三方 Pi 包（pi packages）实现。这种设计让每个用户都能按自己的工 作流定制 Pi，而不需要 fork 或修改 Pi 内部代码。

项目采用 lockstep versioning：所有包共享同一个版本号，patch 版本用于修复和新增，minor 版本用于破坏性变更。

Monorepo 结构

Pi 使用 npm workspaces 管理 monorepo，根 package.json 的 workspaces 字段定义了工作区：

"workspaces": [
    "packages/*",
    "packages/coding-agent/examples/extensions/with-deps",
    "packages/coding-agent/examples/extensions/custom-provider-anthropic",
    "packages/coding-agent/examples/extensions/custom-provider-gitlab-duo",
    "packages/coding-agent/examples/extensions/sandbox",
    "packages/coding-agent/examples/extensions/gondolin"
]

核心包含 5 个包，位于 packages/ 目录下：

包名 npm 包名 职责
tui @earendil-works/pi-tui 终端 UI 框架，提供差分渲染和同步输出
ai @earendil-works/pi-ai 统一多提供商 LLM API，含模型发现和认证
agent @earendil-works/pi-agent-core 通用 Agent 运行时，含工具调用和状态管理
coding-agent @earendil-works/pi-coding-agent 交互式编程助手 CLI，用户入口
orchestrator @earendil-works/pi-orchestrator 实验性编排器，多 Agent 协调

构建顺序由根 package.json 的 scripts.build 定义，严格按照依赖拓扑排序：

"build": "cd packages/tui && npm run build && cd ../ai && npm run build && cd ../agent && npm run build && cd ../coding-agent && npm run build && cd ../orchestrator && npm run build"

构建链：tui → ai → agent → coding-agent → orchestrator

依赖详情

  • pi-tui：无 workspace 依赖。外部依赖 get-east-asian-width（东亚字符宽度计算）和 marked（Markdown 解析）。
  • pi-ai：无 workspace 依赖。外部依赖包括各提供商 SDK（openai、@anthropic-ai/sdk、@google/genai 等）和 typebox（运行时类型校验）。
  • pi-agent-core：依赖 @earendil-works/pi-ai（LLM 调用）和 typebox、yaml、ignore。
  • pi-coding-agent：依赖 @earendil-works/pi-agent-core、@earendil-works/pi-ai、@earendil-works/pi-tui。是所有上游包的集成点。
  • pi-orchestrator：依赖 @earendil-works/pi-coding-agent。实验性包，通过 coding-agent 的 SDK 编排多个 Agent 实例。

分层设计

Pi 采用分层架构，从底层终端渲染到顶层多 Agent 编排：

┌─────────────────────────────────────────────────────────────┐
│                     应用层                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              pi-orchestrator                          │   │
│  │              多 Agent 编排（实验性）                    │   │
│  └──────────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              pi-coding-agent                          │   │
│  │     CLI / SDK / RPC 入口，会话管理，扩展系统             │   │
│  └──────────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────────┤
│                     框架层                                   │
│  ┌────────────────────┐  ┌────────────────────────────┐     │
│  │    pi-agent-core   │  │         pi-tui             │     │
│  │  Agent 运行时       │  │   终端 UI 差分渲染          │     │
│  │  工具调用/状态管理   │  │   组件系统/主题            │     │
│  └────────┬───────────┘  └────────────────────────────┘     │
│           │                                                  │
├───────────┼──────────────────────────────────────────────────┤
│           │           基础设施层                              │
│  ┌────────▼───────────┐                                     │
│  │      pi-ai         │                                     │
│  │  统一 LLM API       │                                     │
│  │  30+ 提供商支持      │                                     │
│  │  流式/工具调用/思考  │                                     │
│  └────────────────────┘                                     │
└─────────────────────────────────────────────────────────────┘

各层职责：

  • 基础设施层（pi-ai）：提供统一的 LLM 调用接口，屏蔽不同提供商的 API 差异。支持流式响应、工具调用（function calling）、推理/思考（reasoning/thinking）、跨提供商上下文迁移、OAuth 认证。是整个系统的模型访问基础。

  • 框架层（pi-agent-core + pi-tui）：pi-agent-core 在 pi-ai 之上构建 Agent 运行时，管理对话状态、工具执行循环、事件流、转向（steering）和跟进（follow-up）队列。pi-tui 提供差分渲染的终端 UI 组件系统，包括编辑器、选择列表、Markdown 渲染等组件。

  • 应用层（pi-coding-agent + pi-orchestrator）：pi-coding-agent 集成所有下层能力，提供完整的编程助手体验——交互模式、print 模式、JSON 模式、RPC 模式、会话管理（JSONL 格式的树状分支）、扩展/技能/提示模板/主题系统、Pi 包管理。pi-orchestrator 在 coding-agent SDK 之上构建多 Agent 协调能力（实验性）。

技术栈

技术 用途
TypeScript 5.9 主要开发语言，全栈类型安全
tsgo (@typescript/native-preview) 构建工具，TypeScript 原生编译器
Biome 2.3 Lint、格式化、import 排序
Vitest 4.x 测试框架
esbuild 脚本和示例构建
Husky Git hooks 管理
Node.js >= 22.19.0 运行时

开发流程：

npm install --ignore-scripts  # 安装依赖（跳过 lifecycle scripts）
npm run build                 # 构建所有包
npm run check                 # Lint + 格式化 + 类型检查
./test.sh                     # 运行测试
./pi-test.sh                  # 从源码运行 pi

与 pi.dev 和 RFC 的关系

  • pi.dev：Pi 的项目网站，提供文档、演示和安装指南。安装脚本 https://pi.dev/install.sh 和版本检查 API https://pi.dev/api/latest-version 均托管于此。
  • RFC（rfc.earendil.com）：大型变更的设计讨论在 RFC 中进行，部分公开，位于 https://rfc.earendil.com/keyword/pi/。
  • pi-chat：Slack/聊天自动化和工作流由独立仓库 earendil-works/pi-chat 提供，不属于本 monorepo。
