Pi 端到端数据流

本文档描述从用户输入到模型输出的完整数据流，覆盖交互模式（TUI）、print 模式和 RPC 模式的关键路径。

架构分层

┌──────────────────────────────────────────────────────────────────┐
│                   packages/coding-agent (CLI)                    │
│  main.ts → 模式路由 → AgentSession (会话管理/扩展系统)              │
├──────────────────────────────────────────────────────────────────┤
│                   packages/agent (agent-core)                    │
│  Agent → agentLoop → streamAssistantResponse → tool calls        │
├──────────────────────────────────────────────────────────────────┤
│                      packages/ai (pi-ai)                         │
│  models.stream() → Provider 认证 → API 实现 → LLM                │
└──────────────────────────────────────────────────────────────────┘

1. 模式入口：CLI → 模式路由

packages/coding-agent/src/main.ts 的 main() 函数根据参数和环境决定运行模式：

main(args)
│
├─ appMode === "rpc" ─────────────► runRpcMode(runtime)
│                                    (stdin/stdout JSON-RPC)
│
├─ appMode === "interactive" ─────► new InteractiveMode(runtime)
│                                    interactiveMode.run()
│                                    (TUI 编辑器 + 事件渲染)
│
└─ appMode === "print" ───────────► runPrintMode(runtime, ...)
                                     (一次性输出到 stdout)

模式判定逻辑（resolveAppMode，main.ts 第100行）： - --mode rpc → "rpc" - --mode json → "json" - -p / --print 或非 TTY stdin/stdout → "print" - 其他 → "interactive"

3. LLM 调用流程：streamAssistantResponse

streamAssistantResponse() (agent-loop.ts, 第275行)
│

│   AgentMessage[] → transformContext(messages) → AgentMessage[]
│   用于裁剪旧消息、注入外部上下文
│   (agent.ts 第178行，agent-session.ts 中未设置，由扩展 context 事件处理)
│
├─ 2. convertToLlm (必须)
│   AgentMessage[] → convertToLlm(messages) → Message[]

print/RPC 模式差异

print 模式 (runPrintMode):
  session.prompt(text) → 事件订阅 → 收集输出 → 写入 stdout → exit
  无 TUI 渲染，事件直接转换为文本/JSON 输出

RPC 模式 (runRpcMode):
  stdin JSON-RPC → session.prompt(text) → 事件订阅 → JSON 事件序列化 → stdout
  事件通过 JSON-RPC 通知/响应协议传输给调用方

4. 工具调用流程

executeToolCalls() (agent-loop.ts, 第373行)
│
├─ 模式判定：
│   toolExecution === "sequential" 或工具标记 executionMode: "sequential"
│     → executeToolCallsSequential()
│   否则 → executeToolCallsParallel() (默认)
│
└─ 每个 toolCall 的处理：

   ┌─────────────────────────────────────────────────────────────┐
   │ 1. emit tool_execution_start                                │
   │    { toolCallId, toolName, args }                           │
   │                                                             │
   │ 2. prepareToolCall()                                        │
   │    ├─ 查找 tool 定义                                        │
   │    ├─ prepareArguments() (可选兼容层)                        │
   │    ├─ validateToolArguments() (TypeBox schema 校验)         │
   │    └─ beforeToolCall 钩子 (可 block) ──────────────────┐    │
   │                                                        │    │
   │ 3. AgentSession._installAgentToolHooks() (agent-session.ts, │
   │    第419行)                                                  │
   │    │                                                       │
   │    ├─ beforeToolCall → extensionRunner.emitToolCall()      │
   │    │   触发扩展 tool_call 事件 (可 block 或修改 input)      │
   │    │                                                       │
   │    └─ afterToolCall → extensionRunner.emitToolResult()     │
   │        触发扩展 tool_result 事件 (可修改 result)            │
   │                                                             │
   │ 4. executePreparedToolCall()                                │
   │    └─ tool.execute(toolCallId, args, signal, onUpdate)     │
   │       onUpdate → emit tool_execution_update (进度)          │
   │                                                             │
   │ 5. finalizeExecutedToolCall()                               │
   │    └─ afterToolCall 钩子 (可修改 result/terminate)          │
   │                                                             │
   │ 6. emit tool_execution_end                                  │
   │    { toolCallId, toolName, result, isError }               │
   │                                                             │
   │ 7. 构建 toolResultMessage                                   │
   │    { role: "toolResult", toolCallId, content, ... }         │
   │                                                             │
   │ 8. emit message_start + message_end (toolResultMessage)     │
   └─────────────────────────────────────────────────────────────┘

并行模式下的执行顺序

assistant message (toolCall A, toolCall B, toolCall C)
│
├─ preflight (顺序):
│   tool_execution_start(A) → prepare(A)
│   tool_execution_start(B) → prepare(B)
│   tool_execution_start(C) → prepare(C)
│
├─ execute (并发):
│   A.execute() ─┐
│   B.execute() ─┤ 同时执行
│   C.execute() ─┘
│
├─ tool_execution_end 按完成顺序发射:
│   tool_execution_end(C) ← 先完成
│   tool_execution_end(A) ← 其次
│   tool_execution_end(B) ← 最后
│
└─ toolResult messages 按 assistant source 顺序发射:
    message_start/end(toolResult A)
    message_start/end(toolResult B)
    message_start/end(toolResult C)

5. steering / follow-up 消息队列流程

用户按 Enter（agent 正在 streaming/tool execution）
│
├─ Enter → steering 消息 ───────────────────────────────────────┐
│   1. AgentSession._steeringMessages.push(text)                │
│   2. agent.steer({ role: "user", content, timestamp })        │
│   3. emit queue_update 事件 → TUI 显示排队消息                 │
│                                                               │
│   消息在 inner loop 的步骤 8 被拉取:                            │
│   pendingMessages = getSteeringMessages() → drain 队列         │
│   下个 turn 开始时作为 user message 注入                        │
│                                                               │
├─ Alt+Enter → follow-up 消息 ──────────────────────────────────┤
│   1. AgentSession._followUpMessages.push(text)                │
│   2. agent.followUp({ role: "user", content, timestamp })     │
│   3. emit queue_update 事件                                    │
│                                                               │
│   消息在 outer loop 底部被拉取:                                 │
│   followUpMessages = getFollowUpMessages() → drain 队列        │
│   只有在无 tool calls 且无 steering 消息时才处理                 │
│                                                               │
├─ Escape → 取消排队 ────────────────────────────────────────────┤
│   clearQueue() → 返回 steering + followUp 数组 → 恢复到编辑器   │
│                                                               │
└─ Alt+Up → 取回排队消息 ────────────────────────────────────────┘
    clearQueue() → 恢复到编辑器

队列模式 (QueueMode):
│
├─ "one-at-a-time" (默认): 每次 drain 只取一条消息
│   每条消息处理后 LLM 响应，然后才取下一条
│
└─ "all": 一次性 drain 所有消息，LLM 处理全部

steering 与 follow-up 的区别

特性 steering follow-up
触发时机 当前 assistant turn 完成后 agent 完全停止后
注入位置 inner loop 步骤 8 (pendingMessages) outer loop 底部
是否跳过 tool calls 否，先完成当前 tool calls 是，无 tool calls 无 steering 时才触发
TUI 快捷键 Enter Alt+Enter

6. 扩展生命周期与数据流交叉点

扩展事件在数据流的以下关键点介入（参见 packages/coding-agent/docs/extensions.md 第276-345行）：

用户输入
│
├─ input 事件 (可 intercept/transform/handle)
│   在 AgentSession.prompt() 步骤 2，扩展命令之后、skill/template 展开之前
│
├─ before_agent_start 事件 (可注入消息/修改 systemPrompt)
│   在 AgentSession.prompt() 步骤 7，消息构建完成后、agent.prompt() 调用前
│
agent 循环中:
├─ context 事件 (可修改消息列表)
│   在 streamAssistantResponse 的 transformContext 之后、convertToLlm 之前
│
├─ before_provider_request 事件 (可检查/替换请求 payload)
│   在 convertToLlm 之后、HTTP 请求之前
│
├─ after_provider_response 事件 (状态码 + headers)
│   在 HTTP 响应到达后、stream 消费前
│
├─ tool_call 事件 (可 block 或修改参数)
│   在 tool_execution_start 之后、validateToolArguments 之后、execute 之前
│
├─ tool_result 事件 (可修改结果)
│   在 execute 之后、tool_execution_end 之前
│
└─ message_end 事件 (可替换消息)
    在 assistant/toolResult message 完成后

8. 完整端到端路径总结

TUI 编辑器输入
  → InteractiveMode 预处理 (@文件/!bash/命令路由)
  → AgentSession.prompt()
    → 扩展命令检查 → input 事件 → skill/template 展开
    → streaming? queue : continue
    → 模型/API Key 验证 → 构建消息 → before_agent_start 事件
  → Agent.prompt(messages)
    → runAgentLoop(prompts, context, config, processEvents, signal, streamFn)
      → runLoop()
        → inner loop:
          → streamAssistantResponse()
            → transformContext → convertToLlm → 构建 LLM Context
            → models.stream(model, context, options)
              → Provider 路由 → 认证解析 → HTTP 请求 → LLM
              → 流式事件 (start/text_delta/toolcall_*/done)
            → emit message_start/message_update/message_end
            → Agent.processEvents() → AgentSession._handleAgentEvent()
              → 扩展事件转发 → listener 通知 → 持久化 → TUI 渲染
          → executeToolCalls()
            → prepareToolCall → beforeToolCall 钩子/扩展 tool_call 事件
            → tool.execute() → afterToolCall 钩子/扩展 tool_result 事件
            → emit tool_execution_start/update/end
            → 构建 toolResultMessage → 注入上下文
          → turn_end → prepareNextTurn → pull steering → 继续/停止
        → outer loop: pull follow-up → 继续/agent_end
← 整体架构 pi-ai →
