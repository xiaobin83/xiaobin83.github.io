pi-agent-core 设计文档

npm 包：@earendil-works/pi-agent-core | 源码目录：packages/agent/

概述

@earendil-works/pi-agent-core 是 Pi 代理工具链的状态化代理运行时（agent runtime），提供有状态代理（Agent）、工具执行（tool execution）、事件流（event streaming）和会话持久化（session persistence）等核心能力。它建立在 @earendil-works/pi-ai 之上，向上被 @earendil-works/pi-coding-agent 使用。

该包导出四层 API：

  • 高层 API：Agent 类（packages/agent/src/agent.ts）——有状态封装，管理消息历史、工具注册、事件订阅（steering/follow-up）和生命周期控制（prompt/continue/abort）。
  • 编排层 API：AgentHarness 类（packages/agent/src/harness/agent-harness.ts）——在 Agent 之上提供会话持久化、运行时配置管理、操作锁、compaction、树导航和扩展钩子子系统。注意：AgentHarness 不直接使用 Agent 类，而是直接调用 runAgentLoop()（见 agent-harness.ts 的 executeTurn 方法，第 531 行起）。
  • 低层 API：agentLoop、agentLoopContinue、runAgentLoop、runAgentLoopContinue（packages/agent/src/agent-loop.ts）——无状态的事件流，适合需要自定义事件处理的场景。
  • 代理函数：streamProxy（packages/agent/src/proxy.ts）——将模型请求代理到后端服务，适用于浏览器环境。

Agent 类

源码：packages/agent/src/agent.ts

Agent 是低层 runAgentLoop 的有状态封装。它拥有当前对话记录（transcript），发出生命周期事件，执行工具，并提供队列 API（steering/follow-up）。与低层 agentLoop() 不同，Agent 确保事件监听器的异步处理在工具执行前完成——message_end 处理作为工具预检前的屏障。

状态管理

Agent 内部状态定义在 MutableAgentState（agent.ts 第 60-65 行），是 AgentState（packages/agent/src/types.ts 第 322-347 行）的可变版本。

// types.ts 第 322-347 行
interface AgentState {
  systemPrompt: string;
  model: Model<any>;
  thinkingLevel: ThinkingLevel;
  tools: AgentTool<any>[];
  messages: AgentMessage[];
  readonly isStreaming: boolean;
  readonly streamingMessage?: AgentMessage;
  readonly pendingToolCalls: ReadonlySet<string>;
  readonly errorMessage?: string;
}

tools 和 messages 的 setter 使用 accessor 实现（agent.ts 第 77-93 行），赋值时拷贝顶层数组，防止外部持有引用后意外修改内部状态。pendingToolCalls 使用 Set<string> 跟踪当前正在执行工具调用的 ID。

状态通过 agent.state 暴露，其中 isStreaming、streamingMessage、pendingToolCalls、errorMessage 为 readonly。在流式传输期间，agent.state.streamingMessage 包含当前部分完成的 assistant 消息。agent.state.isStreaming 在运行完全结束（包括 agent_end 订阅者 await 完成）前保持 true。

prompt / continue / abort

  • prompt(input, images?)（agent.ts 第 334-345 行）：接受字符串、AgentMessage 或消息数组。将输入规范化为 AgentMessage[]，然后调用 runPromptMessages()，后者通过 runAgentLoop() 启动低层循环。如果已有活跃运行则抛出错误。
  • continue()（agent.ts 第 348-375 行）：从当前 transcript 恢复。最后一条消息必须是 user 或 toolResult（不能是 assistant）。如果最后是 assistant 消息，会先尝试 drain steering 和 follow-up 队列。
  • abort()（agent.ts 第 310-312 行）：取消当前运行（通过 AbortController.abort()）。不会丢弃队列中的消息。
  • waitForIdle()（agent.ts 第 319-321 行）：等待当前运行和所有 agent_end 监听器完成。
  • reset()（agent.ts 第 324-332 行）：清除 transcript 状态、运行时状态和队列消息。

事件订阅模型

subscribe() 方法（agent.ts 第 241-244 行）注册生命周期监听器。监听器 Promise 按注册顺序 await，且计入当前运行的 settlement。agent_end 是运行的最终事件，但 agent 在所有 agent_end 监听器 settle 后可变为 idle。

事件通过 processEvents() 方法（agent.ts 第 527-574 行）处理：

  1. 根据事件类型更新内部状态（message_start 设置 streamingMessage，message_end 将消息推入 _state.messages 等）
  2. 依次 await 所有注册的监听器

关键设计：message_end 处理在工具预检（beforeToolCall）前完成，因此 beforeToolCall 看到的 agent 状态已包含请求工具调用的 assistant 消息。

事件类型

AgentEvent 定义在 packages/agent/src/types.ts 第 413 行起：

事件 描述
agent_start Agent 开始处理
agent_end 运行的最终事件，awaited 订阅者计入 settlement
turn_start 新 turn 开始（一次 LLM 调用 + 工具执行）
turn_end Turn 完成，携带 assistant 消息和工具结果
message_start 消息开始（user、assistant、toolResult）
message_update 仅 assistant。包含 assistantMessageEvent 和 delta
message_end 消息完成
tool_execution_start 工具开始执行
tool_execution_update 工具流式推送进度
tool_execution_end 工具执行完成

Steering 和 Follow-up 队列

Steering 消息用于在工具运行时中断 agent，follow-up 消息用于在 agent 正常停止后排队追加工作。

  • steer(message)（agent.ts 第 274-276 行）：排队 steering 消息，在当前 assistant turn 完成后注入
  • followUp(message)（agent.ts 第 279-281 行）：排队 follow-up 消息，仅在无工具调用和 steering 消息时才注入
  • 队列模式（agent.ts 第 256-271 行）："one-at-a-time"（默认，每次 drain 只取一条）或 "all"（一次取完所有排队消息）
  • clearSteeringQueue()、clearFollowUpQueue()、clearAllQueues()：清除排队消息

当 turn 完成后检测到 steering 消息时：当前 assistant 消息的所有工具调用已完成 → 注入 steering 消息 → 下一 turn LLM 响应。Follow-up 消息仅在无工具调用且无 steering 消息时才检查。

配置选项

Agent 构造函数的 AgentOptions（agent.ts 第 97-121 行）包含：

  • initialState：初始状态（systemPrompt、model、thinkingLevel、tools、messages）
  • convertToLlm：将 AgentMessage[] 转换为 LLM Message[]
  • transformContext：在 convertToLlm 前裁剪或注入上下文
  • streamFn：自定义流函数（默认 streamSimple）
  • getApiKey：动态 API Key 解析（用于过期 OAuth token）
  • beforeToolCall / afterToolCall：工具钩子
  • prepareNextTurn / prepareNextTurnWithContext：turn 间状态准备
  • steeringMode / followUpMode：队列模式
  • sessionId：用于 provider 缓存的会话 ID
  • thinkingBudgets：token 提供商的思考预算
  • toolExecution：工具执行模式

agent-loop.ts：Turn 循环

源码：packages/agent/src/agent-loop.ts

runLoop 双层循环结构

核心函数 runLoop()（agent-loop.ts 第 155-269 行）实现双层循环：

  • 外层循环：在 follow-up 消息到达时继续（agent 本应停止后注入新消息）
  • 内层循环：处理工具调用和 steering 消息
runLoop:
  while (true) {                              // 外层：follow-up 循环
    while (hasMoreToolCalls || pending) {     // 内层：工具 + steering 循环
      emit turn_start
      drain pending messages (steering)
      streamAssistantResponse -> LLM 调用
      if (tool calls) executeToolCalls
      emit turn_end
      check shouldStopAfterTurn
      drain steering messages -> pending
    }
    drain follow-up messages -> pending
    if (no follow-up) break
  }
  emit agent_end

关键流程：

  1. 首次进入时检查 steering 消息（用户可能在等待时输入）
  2. 注入 pending 消息（steering 或 follow-up）
  3. 调用 streamAssistantResponse() 获取 LLM 响应
  4. 如果有工具调用，执行工具
  5. 发出 turn_end
  6. 检查 shouldStopAfterTurn 回调
  7. 从 getSteeringMessages() 获取新的 steering 消息
  8. 内层循环结束后，从 getFollowUpMessages() 获取 follow-up 消息

消息转换流程

每次 LLM 调用前，消息经过两级转换（streamAssistantResponse，agent-loop.ts 第 275-368 行）：

AgentMessage[] -> transformContext() -> AgentMessage[] -> convertToLlm() -> Message[] -> LLM
                     (可选)                                (必需)
  1. transformContext：裁剪旧消息、注入外部上下文
  2. convertToLlm：过滤 UI-only 消息、将自定义类型转换为 LLM 格式

工具执行：并行 vs 串行

工具执行模式通过 executeToolCalls()（agent-loop.ts 第 373-388 行）控制：

  • parallel（默认）：顺序预检工具调用（preflight），然后并发执行允许的工具。tool_execution_end 按工具完成顺序发出，但持久化的 toolResult 消息按 assistant 源顺序发出
  • sequential：逐个执行工具调用，匹配历史行为

如果任何工具调用目标的工具设置了 executionMode: "sequential"，整个批次强制串行执行，忽略全局设置。

并行执行流程（executeToolCallsParallel，agent-loop.ts 第 451-516 行）：

  1. 顺序预检所有工具调用（验证参数、调用 beforeToolCall）
  2. 并发执行所有已预检的工具（Promise.all）
  3. 每个工具完成后发出 tool_execution_end
  4. 按 assistant 源顺序发出 toolResult 消息

串行执行流程（executeToolCallsSequential，agent-loop.ts 第 395-449 行）：

  1. 逐个预检、执行、完成工具调用
  2. 每个工具完成后立即发出 tool_execution_end 和 toolResult 消息

beforeToolCall / afterToolCall 钩子

  • beforeToolCall（types.ts 第 89-98 行的 BeforeToolCallContext）：在 tool_execution_start 和参数验证后运行。返回 { block: true, reason: "..." } 可阻止工具执行，循环发出一个错误 toolResult。
  • afterToolCall（types.ts 第 101-114 行的 AfterToolCallContext）：在工具执行完成后、tool_execution_end 和最终 toolResult 事件发出前运行。可返回 { terminate: true } 跳过自动 LLM 跟进调用，或返回 { details: {...} } 覆盖 toolResult 的 details。

shouldStopAfterTurn

shouldStopAfterTurn（types.ts 第 117 行起的 ShouldStopAfterTurnContext）在 turn_end 发出后、assistant 响应和工具执行正常完成后运行。如果返回 true，循环发出 agent_end 并退出，不会轮询 steering/follow-up 队列，也不会启动新的 LLM 调用。它不会中止 provider 流、不会取消运行中的工具、不会修改 assistant 消息的 stopReason。

prepareNextTurn

prepareNextTurn（types.ts 第 129 行起的 AgentLoopTurnUpdate）在 turn_end 发出后运行，允许在下一 turn 前更新 context、model 和 thinkingLevel。常用于 save-point 快照刷新。

types.ts：核心类型

源码：packages/agent/src/types.ts

AgentMessage

// types.ts 第 314 行
type AgentMessage = Message | CustomAgentMessages[keyof CustomAgentMessages];

AgentMessage 是 @earendil-works/pi-ai 中 Message（UserMessage | AssistantMessage | ToolResultMessage）与自定义消息类型的联合。通过 TypeScript declaration merging 扩展：

declare module "@earendil-works/pi-agent-core" {
  interface CustomAgentMessages {
    notification: { role: "notification"; text: string; timestamp: number };
  }
}

LLM 只理解 user、assistant、toolResult 三种角色。convertToLlm 函数在每次 LLM 调用前过滤和转换消息，将自定义类型映射为 LLM 兼容格式。

AgentTool

// types.ts 第 371-394 行
interface AgentTool<TParameters extends TSchema = TSchema, TDetails = any> extends Tool<TParameters> {
  label: string;
  prepareArguments?: (args: unknown) => Static<TParameters>;
  execute: (
    toolCallId: string,
    params: Static<TParameters>,
    signal?: AbortSignal,
    onUpdate?: AgentToolUpdateCallback<TDetails>,
  ) => Promise<AgentToolResult<TDetails>>;
  executionMode?: ToolExecutionMode;
}

关键设计：

  • execute 在失败时抛出错误（不返回错误消息作为 content），错误被 agent 捕获并以 isError: true 报告给 LLM
  • onUpdate 回调用于流式推送部分执行结果（触发 tool_execution_update 事件）
  • 返回 terminate: true 提示 agent 在当前批次后停止（仅当批次中所有工具结果都设置时才生效）
  • executionMode 可覆盖单个工具的并行/串行行为

AgentState

AgentState（types.ts 第 322-347 行）是 agent 的公共状态接口。通过 agent.state 访问。赋值 agent.state.tools = [...] 或 agent.state.messages = [...] 会拷贝顶层数组后存储。

AgentEvent

AgentEvent（types.ts 第 413 行起）是所有 agent 生命周期事件的联合类型，共 10 种事件类型。每个事件携带不同的 payload：

  • agent_start / agent_end：运行边界，agent_end 携带 messages: AgentMessage[]
  • turn_start / turn_end：turn 边界，turn_end 携带 message 和 toolResults
  • message_start / message_update / message_end：消息生命周期
  • tool_execution_start / tool_execution_update / tool_execution_end：工具执行生命周期

CustomAgentMessages 声明合并

CustomAgentMessages 接口（types.ts 第 305-309 行）是一个空接口，通过 TypeScript declaration merging 扩展。应用可以添加自定义消息角色，无需修改核心类型。AgentMessage 联合类型自动包含所有扩展的消息类型。

其他关键类型

  • AgentContext（types.ts 第 397-404 行）：传递给低层循环的上下文快照，包含 systemPrompt、messages、tools
  • AgentLoopConfig（types.ts 第 140 行起）：低层循环配置，包含 model、convertToLlm、transformContext、toolExecution、beforeToolCall、afterToolCall、shouldStopAfterTurn、getSteeringMessages、getFollowUpMessages、prepareNextTurn 等
  • StreamFn（types.ts 第 27-31 行）：自定义流函数类型，签名为 (model, context, options?) => AssistantMessageEventStream
  • QueueMode（types.ts 第 49 行）："all" 或 "one-at-a-time"
  • ToolExecutionMode（types.ts 第 41 行）："sequential" 或 "parallel"
  • AgentToolResult<T>（types.ts 第 350-360 行）：工具执行结果，包含 content、details 和可选的 terminate
  • ThinkingLevel（types.ts 第 289 行）："off" | "minimal" | "low" | "medium" | "high" | "xhigh"

AgentHarness 子系统

源码：packages/agent/src/harness/agent-harness.ts

AgentHarness 是低层 agent loop 之上的编排层。它拥有会话持久化、运行时配置管理、资源解析、操作锁和扩展钩子语义。详细设计文档见 packages/agent/docs/agent-harness.md，本节聚焦核心架构。

与 Agent 类的关系

AgentHarness 不直接使用 Agent 类。 它直接调用 runAgentLoop()（从 agent-loop.ts 导入），自己管理运行生命周期、abort controller、队列 draining、provider 流配置、事件归约、会话持久化、pending write 刷新和 save-point 快照。具体见 agent-harness.ts 的 executeTurn() 方法（第 531 行起）。

会话持久化

Harness 通过 Session 类（packages/agent/src/harness/session/session.ts 第 82 行）管理会话。Session 封装了 SessionStorage 接口，提供：

  • appendMessage(message)（session.ts 第 132 行）：追加消息条目，自动生成 id 和 parentId
  • getBranch(fromId?)（session.ts 第 109 行）：获取从叶节点到根的路径
  • buildContext()（session.ts 第 114 行）：从分支条目构建 AgentContext
  • appendCompaction(summary, firstKeptEntryId, ...)（session.ts 第 173 行）：追加 compaction 条目
  • moveTo(entryId, summary?)（session.ts 第 247 行）：移动叶指针并可选地追加分支摘要

在 harness 中，会话写入分为两种情况：

  • idle 阶段：直接持久化到 session
  • busy 阶段：排队为 pendingSessionWrites，在 save point、settlement 或 failure cleanup 时刷新

Turn 快照

Turn 快照（AgentHarnessTurnState，agent-harness.ts 第 141-155 行）是单次 LLM turn 使用的具体状态，由 createTurnState() 创建，包含：

  • 持久化会话消息（messages）
  • 已解析的资源和 system prompt
  • model、thinking level、所有工具、活跃工具
  • stream options 和派生的 session id

静态选项值直接使用。system prompt provider 回调每次 createTurnState() 调用时执行一次。该 turn 的所有逻辑使用同一快照。

阶段状态机

Harness 有显式的阶段（AgentHarnessPhase，packages/agent/src/harness/types.ts 第 492 行）：

type AgentHarnessPhase = "idle" | "turn" | "compaction" | "branch_summary" | "retry";

结构性操作（prompt、skill、promptFromTemplate、compact、navigateTree）要求 phase === "idle"，在第一个 await 前同步设置阶段。忙碌时启动另一个结构性操作会以 AgentHarnessError 代码 "busy" 拒绝。

以下操作在 turn 期间允许：

  • steer、followUp、nextTurn、abort
  • 运行时配置 setter（影响下一 turn 快照，不改变当前 provider 请求）

Save Points

Save point 在 assistant turn 及其 toolResult 消息完成后发生。在 save point 时 harness：

  1. 在 agent 发出的消息之后刷新 pending session writes
  2. 如果低层循环可能继续，创建新的 turn 快照
  3. 在下一个 provider 请求前应用新的 context/model/thinking-level/stream-options/session-id 状态

这使得 turn 期间对 model、thinking level、tool、resource、stream option 和 system prompt 的更改能影响同一运行中的下一个 turn，同时绝不改变正在进行的 provider 请求。

Pending Writes

会话写入在 busy 期间排队为 pending session writes（agent-harness.ts 的 pendingSessionWrites 数组）。Pending writes 基于 session entry 形状，不包含生成的字段（id、parentId、timestamp）。它们始终被持久化，在 save points、操作 settlement 和 failure cleanup 时刷新。

Agent 发出的消息在 message_end 时持久化以保持 transcript 顺序。Pending extension/session writes 在这些消息之后的 save points 刷新。

Provider Hooks

Harness 提供三个 provider 钩子（agent-harness.ts 第 251-292 行）：

  • before_provider_request：在 provider 请求前运行，可修改 stream options（支持 patch 链式调用和显式字段删除）
  • before_provider_payload：在 payload 发送前运行，可修改请求体
  • after_provider_response：在 provider 响应后运行

Compaction

Compaction 操作（agent-harness.ts 第 686-730 行）在 idle 时执行：

  1. 获取分支条目 → 调用 prepareCompaction() 准备
  2. 发出 session_before_compact 钩子（扩展可提供自定义 compaction 结果）
  3. 调用 compact() 执行实际压缩或使用钩子提供的结果
  4. 追加 compaction 条目到 session
  5. 发出 session_compact 事件

树导航

树导航操作（agent-harness.ts 第 732-827 行）在 idle 时执行：

  1. 收集从旧叶到目标的分支条目
  2. 发出 session_before_tree 钩子（扩展可提供自定义摘要）
  3. 可选地使用 LLM 生成分支摘要
  4. 调用 session.moveTo() 移动叶指针并追加分支摘要
  5. 发出 session_tree 事件

Session 子系统

JSONL 存储

会话以 JSONL（JSON Lines）格式持久化。每行是一个独立的 JSON 条目。实现位于：

  • JsonlSessionStorage（packages/agent/src/harness/session/jsonl-storage.ts 第 171 行）：底层存储，管理文件的读写和条目追加
  • JsonlSessionRepo（packages/agent/src/harness/session/jsonl-repo.ts 第 38 行）：仓库层，管理会话的创建、打开、列出、删除和 fork

文件系统操作通过 FileSystem 接口（packages/agent/src/harness/types.ts 第 252 行）抽象，支持 readTextFile、appendFile、writeFile、listDir、exists、createDir、remove 等操作。

树结构（id / parentId）

每个会话条目都有 id 和 parentId 字段，形成树结构：

  • id：条目的唯一标识符（UUIDv7）
  • parentId：父条目 ID（null 表示根节点）

条目类型包括：

类型 描述
message LLM 消息（user、assistant、toolResult）
thinking_level_change 思考级别变更
model_change 模型变更
active_tools_change 活跃工具集变更
compaction 上下文压缩点
branch_summary 分支摘要
custom_message 自定义消息
label 用户标签（书签）
session_info 会话名称
custom 通用自定义条目

叶指针（leaf pointer）通过 leaf 条目持久化。setLeafId() 不是仅内存操作，而是追加持久化条目，其 targetId 为活跃树叶或 null（根）。重新打开存储时必须从最新持久化的 leaf 条目重建当前叶。

UUIDv7

条目 ID 使用 UUIDv7 生成（createSessionId()，packages/coding-agent/src/core/session-manager.ts 第 204 行）。UUIDv7 是时间排序的 UUID，便于按创建时间排序和索引。

SessionManager（coding-agent 层）

SessionManager（packages/coding-agent/src/core/session-manager.ts 第 780 行）是 coding-agent 层的会话管理器，在 Session 类之上提供更丰富的功能：

  • 会话列表、创建、打开、fork
  • 内存索引（byId Map）加速查询
  • label 管理
  • 分支操作（branch(fromId)）
  • 树遍历

上下文构建

buildSessionContext()（packages/agent/src/harness/session/session.ts 第 22 行）从分支条目构建 SessionContext：

  1. 遍历条目提取 model、thinking level、active tools 和 compaction 信息
  2. 如果有 compaction 条目，从 firstKeptEntryId 开始收集保留的消息，并前置 compaction summary 消息
  3. 否则收集所有消息条目

Compaction 子系统

源码：packages/agent/src/harness/compaction/compaction.ts

prepareCompaction

prepareCompaction()（compaction.ts 第 545 行）准备 compaction 输入：

  1. 如果最后条目已是 compaction，返回 undefined（无需再次压缩）
  2. 找到上一次 compaction 的位置和摘要
  3. 计算当前 token 数
  4. 调用 findCutPoint() 找到裁剪点
  5. 提取要摘要的消息和 turn prefix 消息
  6. 提取文件操作信息

返回 CompactionPreparation（compaction.ts 第 525 行），包含 firstKeptEntryId、messagesToSummarize、turnPrefixMessages、isSplitTurn、tokensBefore、previousSummary、fileOps 和 settings。

分支摘要（Branch Summarization）

generateBranchSummary()（packages/agent/src/harness/compaction/branch-summarization.ts）在树导航时生成分支变更摘要：

  1. 收集从旧叶到新叶的条目
  2. 使用 LLM 生成结构化摘要（包含 readFiles、modifiedFiles 等）
  3. 摘要作为 branch_summary 条目持久化

serializeConversation

serializeConversation()（packages/agent/src/harness/compaction/utils.ts 第 91 行）将 LLM 消息序列化为纯文本，用于 compaction 和 branch summary 的 LLM 输入。它处理各种消息角色（user、assistant、toolResult），提取文本内容，并应用截断以避免超出 token 限制。

compact 主函数

compact()（compaction.ts 第 630 行）执行实际的压缩：

  1. 调用 generateSummary() 生成主摘要（如果有 previousSummary 则使用更新提示）
  2. 如果 isSplitTurn，调用 generateTurnPrefixSummary() 生成 turn 前缀摘要
  3. 返回 CompactionResult，包含 summary、firstKeptEntryId、tokensBefore 和可选的 turnPrefixSummary

proxy.ts（streamProxy）

源码：packages/agent/src/proxy.ts

streamProxy()（proxy.ts 第 116 行）将模型请求代理到后端服务，适用于浏览器环境。它实现 StreamFn 接口：

  1. 构造 ProxySerializableStreamOptions（可序列化的选项子集）
  2. 通过 fetch() 向 ${proxyUrl}/api/stream 发送 POST 请求
  3. 使用 Authorization: Bearer ${authToken} 认证
  4. 解析 SSE（Server-Sent Events）格式的响应流
  5. 通过 processProxyEvent()（proxy.ts 第 238 行）将代理事件转换为 AssistantMessageEvent 流

代理事件类型与 AssistantMessageEvent 一一对应（start、text_start/delta/end、thinking_start/delta/end、toolcall_start/delta/end、done、error）。

使用示例：

const agent = new Agent({
  streamFn: (model, context, options) =>
    streamProxy(model, context, {
      ...options,
      authToken: "...",
      proxyUrl: "https://your-server.com",
    }),
});
