构建与发布

本文档描述 pi-monorepo 的构建系统、检查管道、发布流程及供应链安全策略。

Monorepo 工具链

项目基于 npm workspaces 管理多包结构，workspace 定义在根 package.json 的 workspaces 字段：

  • packages/*（tui、ai、agent、coding-agent、orchestrator）
  • packages/coding-agent/examples/extensions/ 下的扩展包

核心工具链选型：

工具 用途 配置文件
npm workspaces 包管理与依赖安装 package.json (workspaces)
Biome 代码格式化和 lint biome.json
tsgo (@typescript/native-preview) TypeScript 类型检查 tsconfig.base.json
vitest 单元测试 各包内 vitest.config.ts
esbuild 浏览器兼容性冒烟测试 通过 scripts/check-browser-smoke.mjs 调用
husky Git hooks 管理 .husky/pre-commit

所有外部直接依赖使用精确版本号，通过 scripts/check-pinned-deps.mjs 强制执行。内部 workspace 包（@earendil-works/pi-*）保持版本范围引用。

构建流程

构建脚本定义在根 package.json 的 scripts.build：

cd packages/tui && npm run build &&
cd ../ai && npm run build &&
cd ../agent && npm run build &&
cd ../coding-agent && npm run build &&
cd ../orchestrator && npm run build

构建顺序由依赖关系决定：

  1. tui — 终端 UI 库，无内部依赖
  2. ai — 多供应商 LLM API 统一层，无内部依赖
  3. agent — Agent 运行时，依赖 ai
  4. coding-agent — 交互式编码 agent CLI，依赖 tui、ai、agent
  5. orchestrator — 编排层，依赖 coding-agent

各包构建命令通过 npm run clean 清理旧产物后执行 tsc -p tsconfig.build.json 编译 TypeScript。使用 erasableSyntaxOnly 模式，Node.js 可直接运行编译产物无需额外转译。

检查管道

npm run check 执行 7 步检查，按顺序串联：

biome check --write --error-on-warnings . &&
npm run check:pinned-deps &&
npm run check:ts-imports &&
npm run check:shrinkwrap &&
npm run check:install-lock:coding-agent &&
tsgo --noEmit &&
npm run check:browser-smoke

1. Biome 格式化和 Lint (biome check --write --error-on-warnings)

基于 biome.json 配置，对 packages/*/src/**/*.ts、packages/*/test/**/*.ts 和 packages/coding-agent/examples/**/*.ts 执行自动格式化和 lint 检查。--write 自动修复可修复问题，--error-on-warnings 将 warning 升级为 error。排除文件包括 models.generated.ts、*.models.ts 和 test-sessions.ts。

2. 依赖版本锁定 (check:pinned-deps)

scripts/check-pinned-deps.mjs 遍历所有 package.json 文件，验证外部直接依赖（非 @earendil-works/pi-* 前缀、非本地协议）使用精确版本号（x.y.z 格式）。不满足条件时报错退出。

3. TypeScript 相对导入检查 (check:ts-imports)

scripts/check-ts-relative-imports.mjs 使用 TypeScript 编译器 API 解析所有 .ts 文件，检查相对导入是否使用 .js 扩展名——这是 Node.js ESM 的运行时要求。不符合规范的导入路径报错。

4. Shrinkwrap 一致性 (check:shrinkwrap)

scripts/generate-coding-agent-shrinkwrap.mjs --check 验证 packages/coding-agent/npm-shrinkwrap.json 与当前 package-lock.json 一致。shrinkwrap 从根 lockfile 提取 coding-agent 及其传递依赖，确保发布的 CLI 包锁定所有传递依赖版本。

5. Install Lock 一致性 (check:install-lock:coding-agent)

scripts/generate-coding-agent-install-lock.mjs --check 验证 packages/coding-agent/install-lock/ 下的 package.json 和 package-lock.json 与当前状态一致。该 lockfile 用于 pi update --self 自更新功能，确保自更新时安装的依赖版本与发布时一致。

6. TypeScript 类型检查 (tsgo --noEmit)

使用 @typescript/native-preview（基于 Go 的 TypeScript 编译器）对所有包执行完整类型检查。配置继承自 tsconfig.base.json，启用 strict、erasableSyntaxOnly 等严格选项。--noEmit 确保只做类型检查不生成文件。

7. 浏览器兼容性冒烟测试 (check:browser-smoke)

scripts/check-browser-smoke.mjs 使用 esbuild 以浏览器平台目标打包 scripts/browser-smoke-entry.ts。如果打包失败（通常因 Node.js 专用 API 被引用到浏览器路径），则报错。这确保 packages/ai 和 packages/web-ui 的代码保持浏览器兼容性。

Pre-commit Hook

.husky/pre-commit 在每次提交前执行：

  1. 运行 scripts/check-lockfile-commit.mjs 检查 lockfile 提交——除非设置 PI_ALLOW_LOCKFILE_CHANGE=1，否则阻止 package-lock.json 被意外提交
  2. 执行完整的 npm run check
  3. 如果暂存文件包含 packages/ai/*、packages/web-ui/*、package.json 或 package-lock.json，额外执行 npm run check:browser-smoke
  4. 重新暂存被 Biome 格式化修改的文件

CI 工作流

ci.yml — 主 CI 流水线

触发条件：main 分支的 push 和 PR。

  • 单 job (build-check-test)，使用 ubuntu-latest
  • 安装系统依赖（canvas 所需的 cairo、pango、jpeg、gif、rsvg，以及 fd、ripgrep）
  • npm ci --ignore-scripts 安装依赖
  • 执行 npm run build -> npm run check -> npm test
  • 使用 concurrency group 取消同一 ref 的重复运行

build-binaries.yml — 二进制构建与发布

触发条件：推送 v* 标签，或手动 workflow_dispatch。

四个 job 按序执行：

  1. build — 构建 Bun 二进制文件（darwin-arm64/x64、linux-arm64/x64、windows-arm64/x64），生成 SHA256SUMS，打包为 GitHub Release 产物
  2. stage-github-release — 创建 draft GitHub Release 并上传二进制资产
  3. publish-npm — 通过 GitHub Actions OIDC 的 npm trusted publishing 发布所有包到 npm，使用 environment: npm-publish 保护
  4. publish-github-release — 将 draft release 发布为正式 release
  5. cleanup-draft-github-release — 如果上述任一步骤失败，清理 draft release

npm-audit.yml — 定期安全审计

触发条件：每日定时 (37 7 * * *) + 手动触发。

  • npm audit --omit=dev --audit-level=moderate 审计生产依赖
  • npm audit signatures --omit=dev 验证 registry 签名完整性

发布流程

发布由 scripts/release.mjs 编排，支持三种 bump 类型：

npm run release:patch   # 修复和新增
npm run release:minor   # 破坏性变更
npm run release:major   # 预留

发布步骤（scripts/release.mjs）

  1. 检查工作区 — 确保无未提交更改
  2. 版本 bump — 调用 npm run version:<type> 更新所有 workspace 包的版本号，通过 scripts/sync-versions.js 同步，刷新 package-lock.json
  3. CHANGELOG 更新 — 将各包 CHANGELOG.md 的 ## [Unreleased] 替换为 ## [x.y.z] - YYYY-MM-DD
  4. 重新生成产物 — 运行 generate-models（ai 包）、shrinkwrap:coding-agent、install-lock:coding-agent
  5. 运行检查 — 执行 npm run check 确保所有产物一致
  6. 提交和打标签 — git commit -m "Release vX.Y.Z" 和 git tag vX.Y.Z
  7. 添加下周期 Unreleased — 在各 CHANGELOG 头部插入新的 ## [Unreleased] 节
  8. 提交 CHANGELOG 更新 — git commit -m "Add [Unreleased] section for next cycle"
  9. 推送 — git push origin main 和 git push origin vX.Y.Z

CI 自动发布

推送 vX.Y.Z 标签后，build-binaries.yml 自动触发：

  • 构建多平台 Bun 二进制文件
  • publish-npm job 通过 GitHub Actions OIDC 的 npm trusted publishing（无需本地 npm publish、OTP 或 WebAuthn）
  • scripts/publish.mjs 是幂等的——跳过已在 npm 上存在的包版本
  • 发布成功后，draft GitHub Release 转为正式 release

发布前本地冒烟测试

scripts/local-release.mjs 在 repo 外部创建隔离的测试环境：

npm run release:local -- --out /tmp/pi-local-release --force

流程： 1. 运行 npm run check 2. 清理并构建所有包 3. 为每个包创建 npm tarball 4. 构建当前平台的 Bun 二进制 5. 创建隔离的 npm 安装（使用 file: 协议引用 tarball） 6. 创建隔离的 Bun 安装

测试要点：从 /tmp 目录运行隔离安装的 pi 命令，验证启动、模型列表、交互模式和真实推理均正常。

供应链安全

依赖版本锁定

  • .npmrc 配置 save-exact=true，确保 npm install <pkg> 自动使用精确版本
  • .npmrc 配置 min-release-age=2，避免安装发布不足 2 天的依赖包（防御供应链攻击窗口）
  • scripts/check-pinned-deps.mjs 在 npm run check 中验证所有外部依赖使用精确版本
  • 内部 workspace 包使用 ^ 范围引用，保持开发灵活性

Lockfile 管理

  • package-lock.json 是依赖真源（ground truth）
  • .husky/pre-commit 通过 scripts/check-lockfile-commit.mjs 阻止意外提交 lockfile 变更
  • 需要提交 lockfile 变更时设置 PI_ALLOW_LOCKFILE_CHANGE=1
  • CI 使用 npm ci --ignore-scripts 安装，确保依赖与 lockfile 完全一致

Shrinkwrap 与 Install Lock

  • packages/coding-agent/npm-shrinkwrap.json 从根 lockfile 生成，锁定 CLI 发布包的所有传递依赖
  • packages/coding-agent/install-lock/ 下的 lockfile 用于 pi update --self 自更新
  • 两个生成脚本均包含 allowedInstallScriptPackages 白名单，新增含 lifecycle script 的依赖会导致检查失败，需显式审核后加入白名单

--ignore-scripts 策略

  • 本地开发安装：npm install --ignore-scripts
  • CI 安装：npm ci --ignore-scripts
  • 用户安装文档：npm install --ignore-scripts @earendil-works/pi-coding-agent
  • pi update --self：内部使用 --ignore-scripts
  • 生命周期脚本仅在白名单明确允许后生效

安全审计

npm-audit.yml 每日定时运行： - 生产依赖漏洞审计（moderate 级别及以上） - Registry 签名验证

npm 发布安全

  • 使用 GitHub Actions OIDC trusted publishing，无需长期令牌
  • 发布流程由 environment: npm-publish 保护，需仓库管理员审批
  • 所有包发布时附带 --provenance 声明，提供可验证的构建来源
