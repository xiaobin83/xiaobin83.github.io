开发贡献指南

本文档是 CONTRIBUTING.md 的补充，提供 pi monorepo 的技术开发细节（环境搭建、项目结构、测试、调试等），不替代 CONTRIBUTING.md。贡献流程、issue/PR 质量门禁、lgtm/lgtmi 机制等参见 CONTRIBUTING.md；代码风格、git 规则、发布流程等参见 AGENTS.md。

开发环境搭建

前置要求

  • Node.js >= 22.19.0
  • npm（随 Node.js 提供）

克隆并安装

git clone https://github.com/earendil-works/pi-mono
cd pi-mono
npm install --ignore-scripts
npm run build

--ignore-scripts 禁止依赖的生命周期脚本执行。pi 的正常安装和构建不需要生命周期脚本。

从源码运行

./pi-test.sh                 # 从仓库根目录运行 pi（可从任意目录执行）
./pi-test.sh --no-env        # 清除所有 API key 环境变量后运行

pi-test.sh 使用 tsx 直接执行 TypeScript 源码，无需构建即可运行交互式 pi。

常用命令

命令 说明
npm run build 按依赖顺序构建所有包
npm run check 完整检查：Biome 格式化 + lint、依赖固定检查、TypeScript 类型检查、shrinkwrap 校验等
./test.sh 运行非 LLM 测试（清除所有 API key 环境变量后执行）
npm test 运行所有测试（含 LLM 依赖的测试，需要 API key）

项目结构

pi 是 npm workspaces monorepo，包含 5 个包：

pi-mono/
├── packages/
│   ├── ai/                # @earendil-works/pi-ai — 统一多 provider LLM API
│   ├── agent/             # @earendil-works/pi-agent-core — Agent 运行时（工具调用、状态管理）
│   ├── tui/               # @earendil-works/pi-tui — 终端 UI 库（差异渲染）
│   ├── coding-agent/      # @earendil-works/pi-coding-agent — 交互式 CLI、RPC 模式、SDK
│   └── orchestrator/      # @earendil-works/pi-orchestrator — 编排
├── scripts/               # 构建、检查、发布脚本
├── tsconfig.base.json     # 共享 TypeScript 配置基础
├── biome.json             # Biome 格式化与 lint 配置
├── test.sh                # 非 LLM 测试入口（清除 API key 后运行）
├── pi-test.sh             # 从源码运行 pi
├── AGENTS.md              # 代码规则（人和 agent 共用）
└── CONTRIBUTING.md        # 贡献门禁与流程

包依赖关系

coding-agent → agent → ai，coding-agent → tui。orchestrator 为独立包。构建顺序已按依赖关系编排在 npm run build 中。

代码规范

本节概述核心规则，完整规范见 AGENTS.md。

TypeScript 配置

tsconfig.base.json 开启了严格模式，关键选项：

  • erasableSyntaxOnly: true — 仅允许可擦除语法（Node strip-only 模式），禁止 enum、namespace、参数属性、import = 等需要 JS 编译产出的语法。使用显式字段 + 构造函数赋值替代参数属性。
  • module: "Node16", moduleResolution: "Node16" — ESM 模块解析
  • rewriteRelativeImportExtensions: true — 相对路径导入自动添加 .ts 扩展名

核心规则

  • 禁止 any — 除非绝对必要。使用精确类型。
  • 禁止动态导入 — 不使用 await import()、import("pkg").Type 等内联导入。仅允许顶层静态导入。
  • 禁止硬编码按键检查 — 如 matchesKey(keyData, "ctrl+x")。将默认值添加到 DEFAULT_EDITOR_KEYBINDINGS 或 DEFAULT_APP_KEYBINDINGS 中。
  • 先读文件再改 — 大范围改动前必须完整阅读相关文件，不能依赖搜索片段。
  • 不要通过降级代码来修复类型错误 — 应升级依赖版本。
  • packages/ai/src/models.generated.ts — 不要直接修改此文件。修改 packages/ai/scripts/generate-models.ts 后重新生成。
  • 不要保留向后兼容 — 除非用户明确要求。

Biome 配置

格式化：tab 缩进、宽度 3、行宽 120。Lint 使用推荐规则集，关闭了 noNonNullAssertion、noExplicitAny（Biome 级别，代码仍需避免 any）、noControlCharactersInRegex、noEmptyInterface。

依赖安全

  • 直接外部依赖固定为精确版本，内部 workspace 包使用版本范围。
  • .npmrc 设置了 save-exact=true 和 min-release-age=2。
  • npm install --ignore-scripts / npm ci --ignore-scripts — 不执行生命周期脚本。
  • Pre-commit 阻止 lockfile 提交，除非设置 PI_ALLOW_LOCKFILE_CHANGE=1。
  • packages/coding-agent/npm-shrinkwrap.json 需通过 node scripts/generate-coding-agent-shrinkwrap.mjs 生成。新的含生命周期脚本的依赖需审核并加入 allowlist。

测试策略

非 LLM 测试

./test.sh

test.sh 会： 1. 备份 ~/.pi/agent/auth.json 2. 清除所有 LLM provider 的 API key 环境变量（Anthropic、OpenAI、Google、DeepSeek、Groq 等 30+ 个） 3. 设置 PI_NO_LOCAL_LLM=1 跳过本地 LLM 测试 4. 运行 npm test（执行所有 workspace 的非 LLM 测试） 5. 退出时恢复 auth.json

单独运行测试

# 从仓库根目录（推荐）
./test.sh

# 从具体包目录运行特定测试
node ../../node_modules/vitest/dist/cli.js --run test/specific.test.ts

Harness 测试与 faux provider

packages/coding-agent/test/suite/ 目录下的测试使用 test/suite/harness.ts + faux provider。faux provider 是一个模拟 LLM provider，不调用真实 API，不消耗 token。

  • 回归测试放在 packages/coding-agent/test/suite/regressions/，命名格式：<issue-number>-<short-slug>.test.ts
  • 创建或修改测试文件后，运行测试并迭代直到通过

Vitest 配置

每个包有独立的 vitest.config.ts。不要直接运行完整 vitest suite，因为它包含 e2e 测试，这些测试在检测到 endpoint/auth 环境变量时会激活。使用 ./test.sh 作为安全入口。

Git 提交规范

本节概述核心规则，完整规范见 AGENTS.md 的 Git 章节。

多 session 并发

pi monorepo 可能同时有多个 pi session 在工作，各自修改不同文件。git 操作如果触及不属于你的 unstaged/staged/untracked 文件，会破坏其他 session 的工作。

允许的操作：

  • 仅提交你自己在当前 session 中修改的文件
  • 使用显式路径暂存：git add <path1> <path2>
  • 提交前运行 git status 确认只暂存了自己的文件
  • packages/ai/src/models.generated.ts 可以和你的文件一起提交

禁止的操作：

  • git reset --hard、git checkout .、git clean -fd、git stash（会破坏其他 agent 的工作）
  • git add -A、git add .（会暂存他人的文件）
  • git commit --no-verify（绕过检查）

提交信息格式

{feat,fix,docs}[(ai,tui,agent,coding-agent)]: <简短描述>

feat(tui): add diff rendering support
fix(agent): handle tool call timeout
docs: update contributing guide

Rebase 冲突

  • 仅解决你自己修改的文件中的冲突
  • 如果冲突出现在你没改过的文件中，abort 并询问用户
  • 禁止 force push

提交时机

除非用户明确要求，不要主动提交。

调试

/debug 命令

在 pi 交互模式中输入 /debug（隐藏命令），会将以下内容写入 ~/.pi/agent/pi-debug.log： - 渲染后的 TUI 行（含 ANSI 转义码） - 发送给 LLM 的最后几条消息

TUI 写日志

设置环境变量 PI_TUI_WRITE_LOG=1 可以启用 TUI 渲染日志输出，用于排查终端渲染问题。

交互模式测试（tmux）

从仓库根目录在 tmux 中运行 TUI 进行受控测试：

tmux new-session -d -s pi-test -x 80 -y 24
tmux send-keys -t pi-test "./pi-test.sh" Enter
sleep 3 && tmux capture-pane -t pi-test -p     # 启动后截图
tmux send-keys -t pi-test "your prompt here" Enter
tmux send-keys -t pi-test Escape               # 特殊键（Ctrl+O 用 C-o，依此类推）
tmux kill-session -t pi-test

贡献流程概述

本文档补充技术开发细节。贡献流程（issue/PR 门禁、lgtm/lgtmi、质量要求）详见 CONTRIBUTING.md。关键规则摘要：

  • 新贡献者的 issue 和 PR 默认被 auto-close，维护者每日审查并重新打开有价值的
  • lgtmi：未来 issue 不再被 auto-close；lgtm：未来 issue 和 PR 均不再被 auto-close
  • 提交 PR 前确保 npm run check 和 ./test.sh 均通过
  • 不要手动编辑 CHANGELOG.md，由维护者添加

完整的代码规范、发布流程、Changelog 规则等详见 AGENTS.md。
