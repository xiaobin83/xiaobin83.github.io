pi-orchestrator

⚠️ 实验性包 — 本包处于活跃开发阶段，CLI、API 及行为均不稳定，可能随时变更或移除。详见 packages/orchestrator/README.md。

@earendil-works/pi-orchestrator（源码：packages/orchestrator/）是 Pi 的进程编排层，负责管理 coding-agent 实例的生命周期、RPC 通信代理以及 Radius 云平台集成。

架构概述

orchestrator 采用守护进程 + CLI 客户端模式，通过 Unix 域套接字（orchestrator.sock）通信：

  • 守护进程：orchestrator serve 启动，监听本地 socket，管理所有 coding-agent 子进程
  • CLI 客户端：orchestrator spawn/list/status/stop/rpc/rpc-stream 通过 socket 向守护进程发送请求
  • 进程管理器（supervisor）：OrchestratorSupervisor（packages/orchestrator/src/supervisor.ts）负责 spawn、stop、RPC 代理
  • RPC 代理：每个 coding-agent 实例以 --mode rpc 子进程方式运行，supervisor 通过 stdin/stdout JSONL 协议与子进程通信

IPC 机制

orchestrator 的 IPC 分为两层：

1. CLI ↔ 守护进程（Unix 域套接字 + JSONL）

CLI 客户端（src/ipc/client.ts）通过 sendIpcRequest 连接到 orchestrator.sock，发送单行 JSON 请求，读取单行 JSON 响应后断开。协议定义在 src/ipc/protocol.ts：

  • 请求类型：spawn、list、status、stop、rpc、rpc_stream
  • 每条消息以 \n 分隔（encodeMessage 执行 JSON.stringify(message) + "\n"）

rpc_stream 是特殊的长连接模式：服务器验证实例存在后，保持连接并将后续 JSONL 消息（RpcCommand、RpcExtensionUIResponse）转发到对应的 coding-agent 子进程，同时将子进程的响应和事件（RpcResponse、AgentSessionEvent、RpcExtensionUIRequest）回写到 socket。

2. 守护进程 ↔ coding-agent 子进程（stdin/stdout JSONL）

RpcProcessInstance（src/rpc-process.ts）通过 spawn 启动 coding-agent 子进程，使用子进程的 stdin/stdout 进行 JSONL 通信：

  • 发送 RPC 命令：send(command) 将 RpcCommand JSON 写入子进程 stdin，通过 id 字段匹配响应
  • 接收响应：stdout 逐行解析，type: "response" 的消息匹配到对应 pendingRequests 的 Promise
  • 事件广播：非 response 类型的消息（如 AgentSessionEvent）广播给所有 eventListeners
  • UI 请求转发：type: "extension_ui_request" 的消息通过 uiRequestHandler 回调处理

Supervisor 模式

OrchestratorSupervisor（src/supervisor.ts）是编排层的核心，以单例形式导出（export const supervisor = new OrchestratorSupervisor()）。核心方法：

spawnInstance

spawnInstance({ cwd, label? }) → InstanceRecord
  1. 创建 InstanceRecord（状态 starting，UUID）
  2. 调用 createRpcProcessInstance 启动 coding-agent 子进程
  3. 绑定事件监听器和退出处理器
  4. 调用 syncInstanceRecord 获取 session 信息
  5. 通过 Radius 注册 Pi
  6. 状态转为 online

失败时调用 failSpawn 清理资源并将状态标记为 error → stopped。

stopInstance

stopInstance(instanceId) → InstanceRecord | undefined
  1. 状态标记为 stopping
  2. 清理绑定（事件监听、UI handler）
  3. 断开 Radius Pi 连接
  4. 调用 rpcProcess.dispose()（发送 SIGTERM 并等待退出）
  5. 从 liveInstances 中移除，持久化状态为 stopped

handleRpc

handleRpc(instanceId, command) → RpcResponse | undefined

将 RPC 命令转发到对应实例的子进程。执行后如果命令属于会话元数据变更类型（new_session/switch_session/fork/clone/set_session_name/prompt），自动调用 syncInstanceRecord 刷新持久化的 session 信息。

openRpcStream

openRpcStream(instanceId, onEvent, onUiRequest) → { handleRpc, handleUiResponse, close } | undefined

为指定实例打开 RPC 流，返回句柄供 IPC server 的长连接使用。注册事件监听器和 UI 请求回调，关闭时清理订阅。

recoverAfterRestart

守护进程重启时调用：将 online/starting 状态的实例标记为 stopped，断开所有 Radius Pi 连接。

与 coding-agent 的关系

orchestrator 将 coding-agent 作为子进程管理。RpcProcessInstance 的 getSpawnCommand()（src/rpc-process.ts:50-61）根据运行环境选择启动方式：

  • Bun 二进制：执行 pi --mode rpc（与可执行文件同目录）
  • Node：通过 require.resolve("@earendil-works/pi-coding-agent/rpc-entry") 加载 RPC 入口

子进程通过 stdin/stdout 接收 JSONL 格式的 RpcCommand，返回 RpcResponse 和 AgentSessionEvent。orchestrator 不关心 coding-agent 的内部状态，仅作为透明代理传递 RPC 消息。

Radius 云集成

RadiusPresence（src/radius.ts）负责将 orchestrator 管理的机器和 Pi 实例注册到 Radius 云平台（radius.pi.dev）。

机器注册

守护进程启动时（src/serve.ts:21）调用 radiusPresence.start()：

  1. 调用 POST /machines/register 注册机器（上报 hostname、platform、arch、version）
  2. 持久化到 machine.json
  3. 启动心跳循环，按服务端返回的 heartbeatIntervalMs 定时发送 POST /machines/{id}/heartbeat

Pi 注册

每次 spawnInstance 成功后（src/supervisor.ts:291）调用 radiusPresence.registerPi()：

  1. 调用 POST /pis/register（上报 machineId、label、cwd、sessionId）
  2. 记录返回的 radiusPiId
  3. 启动 Pi 级别心跳循环

心跳容错

心跳采用指数退避 + 随机抖动策略（src/radius.ts:81-88）：

  • 基础间隔 1s，最大 30s
  • 每次失败后延迟 = min(30s, 1000ms × 2^(failureCount-1)) + 随机抖动（最多 25%）
  • 连续 3 次 404 后触发重新注册

Pi 断开时（stopInstance 或意外退出），调用 POST /pis/{id}/disconnect。

认证

Radius 认证通过 ~/.pi/agent/auth.json 中的 OAuth credential 或环境变量 PI_RADIUS_API_KEY（src/radius.ts:130-142）。可通过 PI_RADIUS_URL 和 PI_RADIUS_ORCHESTRATOR_URL 自定义 API 端点。

数据持久化

orchestrator 在 ~/.pi/orchestrator/（可通过 PI_ORCHESTRATOR_DIR 自定义）下维护两个 JSON 文件：

  • machine.json：机器注册信息（id、createdAt、label），由 src/storage.ts 的 loadMachine/saveMachine 管理
  • instances.json：实例列表（id、status、cwd、sessionId、radiusPiId 等），由 loadInstances/saveInstances/upsertInstance/removeInstance 管理

每次实例状态变更都会即时写入 instances.json，确保守护进程重启后可通过 recoverAfterRestart 恢复一致状态。

生命周期

orchestrator serve
  ├─ 创建 socket
  ├─ recoverAfterRestart()     ← 恢复上次状态，标记旧实例为 stopped
  ├─ radiusPresence.start()    ← 注册机器，启动心跳
  ├─ 监听 socket 连接
  │   ├─ spawn  → spawnInstance()  → 启动 coding-agent 子进程 → 注册 Radius Pi
  │   ├─ rpc    → handleRpc()      → 转发命令到子进程 → 返回响应
  │   ├─ stop   → stopInstance()   → 清理子进程 → 断开 Radius Pi
  │   └─ ...
  └─ SIGINT/SIGTERM
      ├─ supervisor.shutdown() ← 停止所有实例
      ├─ radiusPresence.stop() ← 断开机器连接
      └─ 删除 socket
