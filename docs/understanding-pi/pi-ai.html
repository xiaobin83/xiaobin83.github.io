pi-ai: LLM API 层设计文档

@earendil-works/pi-ai（源码目录 packages/ai/）是 Pi 生态的统一 LLM API 层。它提供多 provider 集合管理、自动认证解析、token 和成本追踪、上下文持久化，以及跨 provider 的会话切换能力。该库仅包含支持工具调用（tool calling）的模型——这是 agentic 工作流的必要条件。

概述

pi-ai 的设计核心是解耦 provider 注册与模型调用。使用者构建一个 Models 集合，在其中注册多个 Provider，然后通过集合的 stream()/complete() 方法统一调用——认证解析、模型路由、跨 provider 消息转换均由库内部处理。

核心设计决策：

  • Provider 拥有认证语义：每个 provider 声明自己的认证方式（API key、OAuth、环境变量、ambient credentials），Models 集合在每次请求时自动解析并合并到请求选项中（packages/ai/src/auth/resolve.ts）。
  • API 实现是共享的：不同 provider 可能使用相同的底层协议（如 xAI、Groq、Cerebras 共享 openai-completions），API 实现模块通过 .lazy.ts 包装器实现按需加载（packages/ai/src/api/lazy.ts）。
  • 错误不抛异常：请求失败始终通过流事件（error 类型）或最终 AssistantMessage.stopReason 上报，不会从 stream()/complete() 中抛出（packages/ai/src/types.ts:375）。

分层架构

pi-ai 的调用路径为：Models 集合 -> Provider -> API 实现 -> SDK。

调用方 (stream/complete)
    |
    v
Models 集合 (packages/ai/src/models.ts)
    |-- 持有 Provider 注册表 (Map<string, Provider>)
    |-- 通过 CredentialStore 解析认证
    |-- 合并认证结果到请求选项
    |-- 委托给 Provider.stream() / Provider.complete()

    |-- 拥有模型目录 (getModels())
    |-- 拥有认证方法 (auth: ProviderAuth)
    |-- 持有 API 实现引用 (api: ProviderStreams | Record<Api, ProviderStreams>)
    |-- 支持动态模型列表刷新 (refreshModels())
    |
    v

    |-- 导出 stream() 和 streamSimple()
    |-- 将 Context 转换为 provider 原生格式
    |-- 调用 SDK 发送请求
    |-- 将响应解析为标准化 AssistantMessageEvent 流
    |

Models 集合

Models 接口（packages/ai/src/models.ts:79）是调用方的唯一入口。通过 createModels(options?) 创建可变集合（MutableModels，packages/ai/src/models.ts:291），通过 setProvider() 注册 provider。

集合负责： - 同步模型查找（getModel(provider, id)、getModels(provider?)） - 认证解析（getAuth(model)，packages/ai/src/models.ts:216） - 请求分发（stream() 委托给 provider，packages/ai/src/models.ts:258） - 便捷方法（complete() = stream().result()，streamSimple()/completeSimple() 使用统一 reasoning 参数）

Provider

Provider 接口（packages/ai/src/models.ts:32）是具体运行时单元。每个 provider 拥有独立的 id、名称、模型列表、认证方法和 API 实现。Provider 通过 createProvider() 工厂函数构建（packages/ai/src/models.ts:323），接受 api 参数：可以是单个 ProviderStreams 实现（单 API provider），也可以是一个按 model.api 分发的 map（混合 API provider，如 GitHub Copilot 同时使用 Anthropic 和 OpenAI 格式）。

类型系统

核心类型定义在 packages/ai/src/types.ts，包括模型描述、消息协议、工具定义和流事件协议。

KnownApi（packages/ai/src/types.ts:15）定义 9 种已知 API 协议，Api = KnownApi | (string & {}) 允许自定义扩展。

Context 和 Message

// packages/ai/src/types.ts:439
interface Context {
  systemPrompt?: string;
  messages: Message[];
  tools?: Tool[];
}

Message（packages/ai/src/types.ts:408）是 UserMessage | AssistantMessage | ToolResultMessage 的联合类型。每种消息都携带 timestamp 时间戳。

AssistantMessage（packages/ai/src/types.ts:383）包含 content（TextContent | ThinkingContent | ToolCall 数组）、usage（token 统计和成本）、stopReason（"stop" | "length" | "toolUse" | "error" | "aborted"）、api 和 provider 元信息。

Tool 定义

// packages/ai/src/types.ts:433
interface Tool<TParameters extends TSchema = TSchema> {
  name: string;
  description: string;
  parameters: TParameters;  // TypeBox schema
}

Tool 参数使用 TypeBox schema 定义，提供类型安全的参数校验。TypeBox 的 Type、Static 和 TSchema 从 @earendil-works/pi-ai 重新导出。

流事件协议（12 种）

AssistantMessageEvent（packages/ai/src/types.ts:453）定义 12 种流事件：

事件类型 描述 关键属性
start 流开始 partial: 初始 AssistantMessage
text_start 文本块开始 contentIndex
text_delta 文本增量 delta, contentIndex
text_end 文本块结束 content, contentIndex
thinking_start 思考块开始 contentIndex
thinking_delta 思考增量 delta, contentIndex
thinking_end 思考块结束 content, contentIndex
toolcall_start 工具调用开始 contentIndex
toolcall_delta 工具参数流式传输 delta, partial.content[contentIndex].arguments
toolcall_end 工具调用完成 toolCall: 完整 ToolCall
done 流正常完成 reason ("stop"/"length"/"toolUse"), message
error 错误/中止 reason ("error"/"aborted"), error

不同内容块的事件不保证连续；消费者必须使用 contentIndex 关联增量/结束事件与对应块。

pi-ai 实现 9 种 API 协议，源码在 packages/ai/src/api/ 目录下（共 28 个文件）。

每种 API 实现模块导出两个函数：stream(model, context, options) 和 streamSimple(model, context, options)。stream 接受该 API 的完整选项类型，streamSimple 使用统一 reasoning 参数。

.lazy.ts 延迟加载模式

每个 API 实现都有对应的 .lazy.ts 包装器（如 api/anthropic-messages.lazy.ts），核心机制在 packages/ai/src/api/lazy.ts：

  • lazyStream(model, setup)（packages/ai/src/api/lazy.ts:39）：同步返回 AssistantMessageEventStream，异步执行 setup 函数（auth 解析、延迟模块加载）。setup 失败时流以 error 事件终止。

  • lazyApi(load)（packages/ai/src/api/lazy.ts:63）：包装动态 import 的 API 实现模块为 ProviderStreams。模块在首次 stream() 调用时才加载；主机的 import 缓存自动去重。

Provider 工厂导入 .lazy.ts 版本而非直接导入实现模块，这样 SDK（@anthropic-ai/sdk、openai 等）在 bundler 代码分割下保持在延迟 chunk 中，在首次请求时加载。

跨 Provider 消息转换

packages/ai/src/api/transform-messages.ts 处理跨 provider 会话切换时的消息兼容性转换：

  • 用户消息和工具结果消息保持不变
  • 同一 provider/API 的 assistant 消息原样保留
  • 不同 provider 的 assistant 消息中的 thinking 块转换为带 <thinking> 标签的文本
  • tool call 和普通文本保持不变
  • 不支持图片的模型接收到的图片内容替换为占位文本

认证系统

认证系统源码在 packages/ai/src/auth/ 目录（5 个文件）。

ProviderAuth 结构

每个 provider 必须声明认证（packages/ai/src/auth/types.ts:179）：

interface ProviderAuth {
  apiKey?: ApiKeyAuth;  // API key 认证
  oauth?: OAuthAuth;    // OAuth 认证
}

CredentialStore

CredentialStore（packages/ai/src/auth/types.ts:47）是应用持有的凭证存储抽象，按 providerId 键控，每个 provider 一个凭证：

interface CredentialStore {
  read(providerId: string): Promise<Credential | undefined>;
  modify(providerId: string, fn: (current) => Promise<Credential | undefined>): Promise<Credential | undefined>;
  delete(providerId: string): Promise<void>;
}

modify 是唯一的写路径——每次变更是序列化的 read-modify-write。OAuth token 刷新在 modify 内部执行，因此并发请求和进程不会重复刷新已轮换的 token（packages/ai/src/auth/types.ts:54）。

默认实现 InMemoryCredentialStore（packages/ai/src/auth/credential-store.ts）通过 promise 链实现 per-provider 序列化。应用注入持久化存储（如 coding-agent 的 AuthStorage）。

Credential 类型（packages/ai/src/auth/types.ts:30）是 ApiKeyCredential | OAuthCredential：

interface ApiKeyCredential {
  type: "api_key";
  key?: string;
  env?: ProviderEnv;  // provider-scoped 环境配置
}

envApiKeyAuth

envApiKeyAuth(name, envVars)（packages/ai/src/auth/helpers.ts:9）为标准 API key provider 生成 ApiKeyAuth：stored credential key 优先，其次按顺序尝试环境变量。

认证解析流程

resolveProviderAuth()（packages/ai/src/auth/resolve.ts:40）在每次请求时执行：

  1. 显式 apiKey 选项优先
  2. 检查 CredentialStore 中是否有存储凭证
  3. 存储的 OAuth credential -> 通过 modify 锁刷新 token -> 派生 ModelAuth
  4. 存储的 API key credential -> 合并 credential key 和 ambient 源
  5. 无存储凭证 -> 通过 provider 的 ApiKeyAuth.resolve() 查找环境变量

Ambient 认证

部分 provider 使用非环境变量的 ambient 认证：Amazon Bedrock 解析 AWS credentials（AWS_PROFILE、access key 对、ECS task roles），Vertex AI 解析 gcloud Application Default Credentials。

OAuth

支持 OAuth 的 provider（Anthropic、OpenAI Codex、GitHub Copilot）通过 OAuthAuth 接口（packages/ai/src/auth/types.ts:154）暴露 login、refresh、toAuth 操作。lazyOAuth()（packages/ai/src/auth/helpers.ts:34）包装动态 import，使 Node-only OAuth 流程代码不出现在浏览器 bundle 中。

Provider 工厂模式

每个内置 provider 对应 packages/ai/src/providers/<id>.ts 一个工厂文件（目录共 75 个条目，包括 .models.ts 模型目录文件）。

工厂示例

以 Anthropic provider 为例（packages/ai/src/providers/anthropic.ts）：

export function anthropicProvider(): Provider<"anthropic-messages"> {
  return createProvider({
    id: "anthropic",
    name: "Anthropic",
    baseUrl: "https://api.anthropic.com",
    auth: {
      apiKey: envApiKeyAuth("Anthropic API key", ["ANTHROPIC_OAUTH_TOKEN", "ANTHROPIC_API_KEY"]),
      oauth: lazyOAuth({ name: "Anthropic (Claude Pro/Max)", load: loadAnthropicOAuth }),
    },
    models: Object.values(ANTHROPIC_MODELS),
    api: anthropicMessagesApi(),
  });
}

注册所有 Provider

packages/ai/src/providers/all.ts 提供两个入口：

  • builtinProviders()：返回所有内置 provider 实例的数组（36 个 provider 工厂调用）
  • builtinModels(options?)：创建 Models 集合并注册所有内置 provider，这是最重量级的入口点

对于按需加载的应用，每个 provider 工厂是独立子路径 import：

import { anthropicProvider } from '@earendil-works/pi-ai/providers/anthropic';

createProvider()

createProvider()（packages/ai/src/models.ts:323）是构建 provider 的通用工厂。接受：

  • id、name、baseUrl、headers：基础元数据
  • auth：ProviderAuth（必需——每个 provider 都有认证语义）
  • models：初始模型列表（纯动态 provider 可为空数组）
  • refreshModels：动态 provider 的模型刷新函数
  • api：ProviderStreams 实例或按 model.api 分发的 map（混合 API provider）

对于自定义 provider（如本地 Ollama 或 vLLM 服务），使用 createProvider() + openAICompletionsApi() + envApiKeyAuth() 即可。

模型生成脚本

packages/ai/scripts/generate-models.ts（2104 行）负责从各 provider 的数据源（如 models.dev API）抓取和解析模型信息，生成标准化的 Model 接口数据。

生成流程： 1. 从各 provider 源获取模型列表（models.dev API、NVIDIA NIM API、Cloudflare AI Gateway API 等） 2. 映射 provider 特有字段到标准 Model 接口（pricing 格式、capability flags、model ID 转换） 3. 输出 packages/ai/src/providers/<id>.models.ts（每个 provider 的模型目录）和聚合文件 packages/ai/src/models.generated.ts

脚本同时处理 provider 特有兼容性配置，如 Together AI 的 supportsStore: false、Cloudflare Workers AI 的 base URL 映射等。注意：不要直接修改 models.generated.ts；修改 scripts/generate-models.ts 后重新生成。

图像生成 API (ImagesModels)

图像生成使用独立的 API 表面，镜像 chat/text 侧的设计：ImagesModels 集合持有 ImagesProvider，同步读取，认证通过 owning provider 解析。图像生成是 one-shot API：generateImages() 等待 provider 响应并返回最终 AssistantImages 结果。

核心文件： - packages/ai/src/images.ts：generateImages() 函数，从 images-api-registry.ts 查找 API 实现并委托调用 - packages/ai/src/image-models.ts：getImageModel()、getImageModels()、getImageProviders() 静态目录读取 - packages/ai/src/api/openrouter-images.ts：OpenRouter 图像生成 API 实现

ImagesModel 接口（packages/ai/src/types.ts:698）继承 Model 的大部分字段，额外提供 output: ("text" | "image")[] 表示输出模态。ImagesContext（packages/ai/src/types.ts:413）包含 input: ImagesInputContent[]。目前图像生成仅通过 OpenRouter provider 提供。

旧版兼容 API (compat.ts)

packages/ai/src/compat.ts 保留旧版全局 API 表面：api-dispatch stream()/complete() + env API key 注入、api-registry、生成目录读取、per-API lazy stream 包装器、图像生成。

// 旧用法
