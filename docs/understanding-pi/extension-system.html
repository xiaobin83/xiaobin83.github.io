扩展系统架构

pi 的扩展系统是一套类型安全的插件框架，允许 TypeScript 模块订阅 agent 生命周期事件、注册 LLM 可调用的自定义工具、添加命令/快捷键/CLI 标志，以及注入自定义 UI 组件。扩展通过 jiti 即时编译，无需构建步骤。

架构概览

扩展系统的核心由四个模块组成：

+--------------------------------------------------------------+
|                        ExtensionAPI                         |
|  (扩展工厂函数接收的 API 对象，用于注册和操作)                |
+--------------------------------------------------------------+
|  registerTool  registerCommand  registerShortcut  registerFlag |
|  registerProvider  on(event, handler)  sendMessage  events    |
|  setActiveTools  appendEntry  setSessionName  exec           |
+----------+---------------------------------------------------+
           | 写入
+----------v---------------------------------------------------+
|                       ExtensionRunner                        |
|  (事件分发中心，持有所有 Extension 实例和运行时状态)          |
+--------------------------------------------------------------+
|  emit(event) -> 遍历所有 Extension 的 handlers 并调用         |
|  createContext() -> 创建 ExtensionContext (懒求值 getter)     |
|  createCommandContext() -> 创建 ExtensionCommandContext        |
|  bindCore() / bindCommandContext() / setUIContext()          |
+----------+---------------------------------------------------+
           | 持有
+----------v---------------------------------------------------+
|                     Extension[] (多个)                       |
+--------------------------------------------------------------+
|  handlers: Map<eventType, HandlerFn[]>                       |
|  tools: Map<toolName, RegisteredTool>                        |
|  commands: Map<commandName, RegisteredCommand>               |
|  flags / shortcuts / messageRenderers / entryRenderers       |
+--------------------------------------------------------------+

loader.ts -- 负责发现和加载扩展模块。支持从目录自动发现 (~/.pi/agent/extensions/、.pi/extensions/)、解析 package.json 的 pi 清单字段，以及通过 -e 标志显式加载。使用 jiti 的 virtualModules (Bun 二进制模式) 或 alias (Node.js 开发模式) 提供对 @earendil-works/pi-coding-agent、@earendil-works/pi-agent-core、@earendil-works/pi-tui、typebox 等内置包的零配置导入。

wrapper.ts -- 将 RegisteredTool (扩展注册的工具定义) 包装为 AgentTool (agent-core 的标准工具接口)。核心逻辑只有两行：调用 wrapToolDefinition 并传入 runner.createContext() 作为上下文工厂，确保工具执行时获得最新的 ExtensionContext。

生命周期与事件流

pi 从启动到用户交互再到退出，经历一个严格的事件序列。下图展示完整流程：

pi 启动
  |
  +--> project_trust         <-- 仅 user/global 扩展和 CLI -e 扩展参与
  |                             (返回 yes/no 则跳过内置信任提示)
  |
  +--> session_start         <-- reason: "startup"
  |
  +--> resources_discover    <-- reason: "startup"
      |                          (扩展可贡献额外的 skill/prompt/theme 路径)
      |
      v
+-------------------------------------------------------------+
|                    用户输入循环                              |
|                                                             |
|  input                  <-- 可拦截 (handled)、转换 (transform) |
|    |                        或放行 (continue)                  |
|    +--> (扩展命令检查，命中则直接执行命令)                     |
|    +--> (skill/template 展开)                                |
|    |                                                        |
|    +--> before_agent_start  <-- 可注入 message、修改 systemPrompt|
|    +--> agent_start                                          |
|    +--> message_start / message_update / message_end         |
|    |                                                        |
|    |   +--- turn 循环 (LLM 持续调用工具则重复) ----+         |
|    |   |                                          |         |
|    |   +--> turn_start                             |         |
|    |   +--> context          <-- 可修改消息列表      |         |
|    |   +--> before_provider_request <-- 可替换 payload|        |
|    |   +--> after_provider_response <-- status+headers|       |
|    |   |                                          |         |
|    |   |   LLM 响应，可能调用工具：                 |         |
|    |   |     +--> tool_execution_start             |         |
|    |   |     +--> tool_call      <-- 可 block       |         |
|    |   |     +--> tool_execution_update            |         |
|    |   |     +--> tool_result    <-- 可修改 content  |        |
|    |   |     +--> tool_execution_end               |         |
|    |   |                                          |         |
|    |   +--> turn_end                               |         |
|    |                                                        |
|    +--> agent_end                                            |
|                                                             |
|  用户再次输入 -----------------------------------------------+
|
+--> /new 或 /resume (会话替换)
|     +--> session_before_switch <-- 可 cancel
|     +--> session_shutdown      <-- reason: "new" | "resume"
|     +--> session_start         <-- reason: "new" | "resume"
|     +--> resources_discover
|
+--> /fork 或 /clone
|     +--> session_before_fork   <-- 可 cancel
|     +--> session_shutdown      <-- reason: "fork"
|     +--> session_start         <-- reason: "fork"
|     +--> resources_discover
|
+--> /compact 或自动压缩
|     +--> session_before_compact <-- 可 cancel 或提供自定义摘要
|     +--> session_compact
|
+--> /tree 导航
|     +--> session_before_tree   <-- 可 cancel 或提供自定义摘要
|     +--> session_tree
|
+--> /model 或 Ctrl+P (模型切换)
|     +--> thinking_level_select  <-- (如果模型切换导致 thinking level 变化)
|     +--> model_select
|
+--> 退出 (Ctrl+C, /quit, shutdown())
      +--> session_shutdown      <-- reason: "quit"

事件分类

启动与资源事件: project_trust (项目信任决策，早于项目资源加载)、resources_discover (扩展贡献额外资源路径)

会话事件: session_start (会话创建/加载/重载)、session_shutdown (运行时销毁前的清理)、session_before_switch (会话切换前，可取消)、session_before_fork (分支前，可取消)、session_before_compact (压缩前，可取消或提供自定义摘要)、session_compact (压缩后)、session_before_tree (树导航前，可取消)、session_tree (树导航后)、session_info_changed (会话名称变更)

Agent 事件: before_agent_start (agent 循环前，可注入消息、修改 system prompt)、agent_start / agent_end、turn_start / turn_end、message_start / message_update / message_end、tool_execution_start / tool_execution_update / tool_execution_end

上下文与请求事件: context (LLM 调用前，可修改消息列表)、before_provider_request (请求发出前，可替换 payload)、after_provider_response (响应接收后，流消费前)

工具事件: tool_call (工具执行前，可 block；可修改 event.input)、tool_result (工具执行后，可修改 content/details/isError)

模型与输入事件: model_select (模型切换)、thinking_level_select (思考等级变更)、input (用户输入，可拦截/转换/放行)、user_bash (! 前缀命令，可替换执行操作或结果)

各事件的详细签名、返回值类型和示例参见 extensions.md。

上下文系统

扩展的事件处理函数和工具执行函数接收两类上下文对象。

ExtensionContext

ExtensionContext 是所有事件处理函数和工具 execute 的基础上下文：

interface ExtensionContext {
  ui: ExtensionUIContext;          // UI 交互方法
  mode: "tui" | "rpc" | "json" | "print";  // 当前运行模式
  hasUI: boolean;                  // 是否有对话框能力 (tui/rpc 为 true)
  cwd: string;                     // 当前工作目录
  sessionManager: ReadonlySessionManager;  // 只读会话管理器
  modelRegistry: ModelRegistry;    // 模型注册表
  model: Model<any> | undefined;   // 当前模型
  isIdle(): boolean;               // agent 是否空闲
  isProjectTrusted(): boolean;     // 项目是否受信任
  signal: AbortSignal | undefined; // 当前 abort 信号
  abort(): void;                   // 中断当前 agent 操作
  hasPendingMessages(): boolean;   // 是否有排队消息
  shutdown(): void;                // 优雅关闭 pi
  getContextUsage(): ContextUsage | undefined;  // 当前上下文用量
  compact(options?: CompactOptions): void;      // 触发压缩
  getSystemPrompt(): string;       // 当前 system prompt
}

所有属性通过 getter 懒求值，保证每次访问都返回最新值 (例如 ctx.model 在模型切换后自动更新)。上下文对象在 session_start 之后保持有效，但在会话替换 (newSession、fork、switchSession) 或 reload 后会变为 stale。访问 stale 上下文会抛出错误，提示将后续逻辑移入 withSession 回调。

ExtensionCommandContext

ExtensionCommandContext 扩展了 ExtensionContext，增加了命令处理器专用的会话控制方法：

interface ExtensionCommandContext extends ExtensionContext {
  getSystemPromptOptions(): BuildSystemPromptOptions;
  waitForIdle(): Promise<void>;         // 等待 agent 完成流式输出
  newSession(options?): Promise<{ cancelled: boolean }>;
  fork(entryId, options?): Promise<{ cancelled: boolean }>;
  navigateTree(targetId, options?): Promise<{ cancelled: boolean }>;
  switchSession(sessionPath, options?): Promise<{ cancelled: boolean }>;
  reload(): Promise<void>;
}

这些方法仅通过命令处理器 (registerCommand 的 handler) 提供，不在普通事件处理函数中可用。waitForIdle 确保命令在 agent 空闲后执行；newSession/fork/switchSession 触发会话替换流程。

ReplacedSessionContext

ReplacedSessionContext 进一步扩展 ExtensionCommandContext，仅在 withSession 回调中提供，新增 sendMessage 和 sendUserMessage，允许在新会话中立即发送消息。

工具系统

注册与执行

扩展通过 pi.registerTool() 注册工具。每个工具定义包含：

  • 参数 schema: 使用 TypeBox 定义，自动生成 JSON Schema 供 LLM 使用
  • execute 函数: 接收 toolCallId、已验证的 params、signal (用于取消)、onUpdate (流式输出回调)、ctx (ExtensionContext)
  • 可选渲染器: renderCall 和 renderResult 控制工具在 TUI 中的显示
  • 可选 prompt 提示: promptSnippet 和 promptGuidelines 用于自定义 system prompt 中的工具描述

工具注册后，wrapper.ts 将其包装为 AgentTool，纳入 agent-core 的标准工具执行流程。扩展工具和内置工具在同一工具调用周期中执行，共享并行/串行调度逻辑。

工具优先级规则：同名工具以先注册的为准，因此扩展可以通过注册与内置工具同名的工具来覆盖内置行为 (例如自定义 bash 工具，将命令路由到远程执行)。

工具事件拦截

tool_call 事件在工具执行前触发，允许扩展：

  • 修改参数: 直接修改 event.input (无需返回值)，后续 handler 和工具执行会看到修改后的值
  • 阻止执行: 返回 { block: true, reason: "..." }

tool_result 事件在工具执行后触发，允许扩展修改 content、details、isError。

这两个事件构成了权限门控 (permission gate)、路径保护 (path protection)、命令审计等扩展模式的基础。

UI 扩展

扩展可以通过 ctx.ui 操作 TUI 的各个视觉区域：

+--------------------------------------------+
|  Header (setHeader)                        |
+--------------------------------------------+
|  Widget above editor (setWidget, default)  |
+--------------------------------------------+
|  Messages / Chat area                      |
|    - 消息渲染 (registerMessageRenderer)     |
|    - 条目渲染 (registerEntryRenderer)       |
|    - 工具调用/结果渲染 (renderCall/renderResult) |
+--------------------------------------------+
|  Widget below editor (setWidget, belowEditor) |
+--------------------------------------------+
|  Editor (setEditorComponent)               |
+--------------------------------------------+
|  Footer (setFooter)                        |
|    - Status (setStatus)                    |
|    - Working indicator (setWorkingIndicator)|
+--------------------------------------------+
  • 自定义编辑器: 通过 setEditorComponent 替换输入编辑器 (如 vim 模式)，继承 CustomEditor 并调用 super.handleInput() 可保留应用级快捷键
  • 浮层 (Overlay): ctx.ui.custom() 配合 { overlay: true } 可在现有内容上显示模态组件
  • 主题管理: ctx.ui.setTheme()、ctx.ui.getAllThemes()、ctx.ui.theme 支持动态主题切换和自定义颜色

会话生命周期与会话替换

扩展的运行实例与当前会话绑定。当发生会话替换 (/new、/resume、/fork、/clone) 或 /reload 时：

  1. 触发 session_before_switch 或 session_before_fork (可被取消)
  2. 触发 session_shutdown (扩展应清理资源)
  3. 销毁旧的 ExtensionRunner 和所有 Extension 实例
  4. 重新加载扩展模块，创建新的实例
  5. 触发 session_start (扩展应重新建立状态)
  6. 旧的 ExtensionContext 变为 stale，访问会抛出错误

关键规则: 不要在 newSession/fork/switchSession/reload 之后继续使用捕获的 pi 或 ctx 引用。会话替换后的工作应移入 withSession 回调，使用回调提供的 ReplacedSessionContext。

错误处理与模式行为

错误处理

  • 扩展事件处理器中的异常会被捕获并记录，agent 继续运行
  • tool_call 处理器异常会阻止工具执行 (fail-safe)
  • 工具 execute 中的异常被捕获后以 isError: true 报告给 LLM，执行继续

模式行为

模式 ctx.mode ctx.hasUI 说明
交互模式 "tui" true 完整 TUI，所有 UI 方法可用
RPC 模式 "rpc" true 对话框通过 JSON 协议；custom() 返回 undefined
JSON 模式 "json" false 事件流输出到 stdout；UI 方法为 no-op
Print 模式 "print" false 扩展运行但无法弹出交互提示

使用 ctx.mode === "tui" 守卫 TUI 专用功能 (custom()、组件工厂、终端输入监听)。使用 ctx.hasUI 守卫在 TUI 和 RPC 下均可用的对话框方法 (select、confirm、input、notify)。

扩展加载机制

扩展通过 loader.ts 的 discoverAndLoadExtensions 函数按以下优先级发现：

  1. 项目本地: <cwd>/.pi/extensions/ 下的 .ts/.js 文件或子目录 (含 index.ts 或 package.json 的 pi.extensions 清单)
  2. 全局: ~/.pi/agent/extensions/ 下的同类结构
  3. 显式配置: settings.json 的 extensions 数组和 CLI -e 标志

每个扩展模块的默认导出工厂函数接收 ExtensionAPI，同步或异步初始化均可。异步工厂在 session_start 之前完成，适合一次性远程配置拉取 (如 pi.registerProvider 动态注册模型)。

扩展加载器通过 jiti 的 virtualModules (Bun 二进制) 或 alias (Node.js 开发) 提供对核心包的零配置导入。npm 依赖也受支持：扩展目录下的 node_modules 自动可用，pi package 安装时执行 npm install --omit=dev。

ExtensionRunner.bindCore() 将运行时 action (sendMessage、appendEntry 等) 绑定到共享的 ExtensionRuntime 对象。bindCore 也会刷新扩展加载期间排队的 registerProvider 调用。bindCommandContext() 提供命令处理器专属方法 (newSession、fork 等)。
