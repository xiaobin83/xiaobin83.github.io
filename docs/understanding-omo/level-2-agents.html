第二层 · Agent 系统

OmO 提供 11 个专业 Agent。每个都有独特的专长、优化的模型选择和工具权限。Agent 之间通过 category 分类系统自动匹配最优模型，无需人工干预。

核心 Agent

项目最核心的 6 个 Agent，构成系统的执行骨架。

Agent推荐模型特性
Sisyphus Claude Opus 4.7 / Kimi K2.6 / GLM 5.1 主编排器。Todo 驱动工作流，规划并委派给专家 agent，激进并行执行。32k 思考预算 (thinking budget)。不半途而废。处理所有顶层用户交互。
Hephaestus GPT-5.5 自主深度工作者。给他目标 (goal)，而非配方 (recipe)。探索代码库、研究模式、端到端执行。灵感来自 AmpCode 的 deep 模式。不写无意义的样板代码。
Oracle GPT-5.5 / Gemini 3.1 Pro / Claude Opus 4.7 只读高智商顾问。架构决策、代码审查、复杂调试分析。仅限 read 工具和搜索 MCP。不写代码、不编辑文件、不委派任务。
Librarian GPT-5.4 Mini / Qwen / MiniMax 多仓库分析。文档搜索、开源实现示例查找、深度代码库理解。基于证据回答，引用具体来源。只读。
Explore GPT-5.4 Mini / Qwen / MiniMax 快速代码库探索。上下文感知 grep 和 glob。速度优先于深度分析。只读，不编辑文件。
Multimodal Looker GPT-5.5 / Kimi K2.6 / GLM 4.6v 视觉内容专家。分析 PDF、图片、截图、图表和 UI 设计稿。仅限 read 和 look_at 工具，不修改任何文件。

Sisyphus 是唯一拥有完整工具权限和委派能力的 Agent。所有其他 Agent 通过 Sisyphus 的任务委派被调用——要么直接指定 agent 类型，要么通过 category 间接路由。

规划 Agent

三阶段规划管道：初始分析 (Metis) → 详细规划 (Prometheus) → 审查验证 (Momus)。

Agent推荐模型特性
Prometheus Claude Opus 4.7 / GPT-5.5 战略规划师。访谈模式 (interview mode) — 像真正的工程师一样，在碰代码之前通过迭代提问识别范围、歧义和约束，构建详细可执行计划。
Metis Claude Sonnet 4.6 / Claude Opus 4.7 规划顾问。在计划最终确定前分析隐藏意图、模糊性和 AI 失败模式。提供元级反馈，但不做最终决策。
Momus GPT-5.5 / Claude Opus 4.7 计划审查官。针对清晰度、可验证性和完整性标准验证计划。确保每个计划项都有可测量的成功标准。只读。

编排 Agent

将计划转化为执行的两个抽象层。

Agent推荐模型特性
Atlas Claude Sonnet 4.6 / Kimi K2.6 Todo 编排器。系统执行计划任务，管理 todo 项并协调工作。通过 /start-work 命令激活，从 Prometheus 计划中加载 todo 列表并逐个推进。
Sisyphus-Junior 按 category 自动选择 Category 衍生执行器。当任务以 category 而非 agent 类型委派时，由 Sisyphus-Junior 实例化并承载该 category 对应的模型。模型从不固定，完全由 category 系统自动决定。

模型推荐与回退链

每个 Agent 都有推荐的模型和回退链。当首选模型不可用 (速率限制、故障) 时，系统自动沿链回退。

Agent首选模型思考预算回退链

Category 分类系统

Category 将任务路由到最优模型，是 OmO 的核心抽象。Agent 声明需要什么类型的工作，而非什么模型。系统根据 category 自动选择模型和思考预算。

8 个内置 Category

Category默认模型思考级别适用场景
visual-engineering Gemini 3.1 Pro high 前端、UI/UX、设计系统、CSS/样式、响应式布局、动效、视觉 QA
ultrabrain GPT-5.5 xhigh 硬逻辑推理、复杂架构决策、算法设计、安全审计、数学密集任务
deep GPT-5.5 medium 目标导向自主问题解决、端到端特性开发、代码库范围重构
artistry Gemini 3.1 Pro high 高度创意/艺术任务、设计探索、视觉概念、品牌设计
quick GPT-5.4 Mini low 琐碎任务：单文件修改、错别字修复、导入排序、简单 lint 修复
unspecified-low Claude Sonnet 4.6 medium 低复杂度杂项任务，不需要专门 category 时使用
unspecified-high Claude Opus 4.7 max 高复杂度杂项任务，跨领域、需要最高推理能力
writing Kimi K2.5 medium 文档、技术写作、README、变更日志、API 文档、知识库文章

task() 委派示例

// 用 category 委派任务——模型自动匹配
task(
  category="visual-engineering",
  description="Redesign dashboard layout",
  prompt="Redesign the admin dashboard with responsive grid layout...",
  run_in_background=true          // 后台异步执行
)

// 加载 skill 并委派——category + skill 组合
task(
  category="ultrabrain",
  load_skills=["security-research"],
  description="Audit auth module",
  prompt="Audit authentication flow for security vulnerabilities..."
)

// 直接指定 agent 类型 (绕过 category 路由)
task(
  subagent_type="oracle",
  description="Review architecture",
  prompt="Review the proposed architecture for the new payment service..."
)

category 与 subagent_type 的区别：category 路由到 Sisyphus-Junior 并自动匹配模型——灵活、可配置。subagent_type 直接调用指定 Agent (如 oracle、librarian)——确定、不可覆盖。大多数情况优先使用 category。

后台 Agent 系统

后台 Agent 允许 Sisyphus 同时启动多个专家 agent 并行工作，而非顺序等待。这是 OmO 实现高吞吐的核心机制。

并行执行模型

并发控制

每个 providerID/modelID 键默认最多 5 个并发后台任务。超出限制的任务进入 FIFO 队列等待空闲槽位。

上下文隔离

每个后台任务拥有独立的会话上下文，不污染主 session 的上下文窗口。结果通过 background_output(task_id) 按需取回。

可配置限流

通过 background_task.modelConcurrency 和 providerConcurrency 可分别控制每个模型和每个提供商的并发上限。

完成通知

后台任务完成后，系统通过 <system-reminder> 通知主 session，附带 task_id。主 session 通过 background_output(task_id) 取回结果。

// 启动后台 agent——立即返回 task_id
task(
  subagent_type="explore",
  description="Search auth patterns",
  prompt="Find all authentication-related patterns across the codebase...",
  run_in_background=true           // 关键参数
)
// → 返回: {"task_id": "bg_abc123", "status": "started"}

// 后续主 session 收到完成通知后取结果
background_output(task_id="bg_abc123")
// → 返回: agent 的完整输出

禁止使用 background_cancel(all=true)：只能通过 taskId 逐个取消。全量取消会导致不确定状态。

Team Mode (v4.0)

Team Mode 将 OmO 从"一个 agent + 子 agent"转变为真正的多 agent 协作系统。Lead agent 协调一个并行运行的团队，通过专用工具族通信。默认为关闭状态。

架构概览

Lead + 成员结构

一个 Lead agent 编排最多 8 个并行成员。成员可以通过 kind: "subagent_type" (直接 agent) 或 kind: "category" (通过 Sisyphus-Junior 路由) 声明。

共享延迟确认邮箱

成员之间通过共享的 deferred-ack mailbox 通信。消息写入 mailbox 目录，接收方轮询读取。确认机制防止消息丢失。

共享任务列表

JSONL 格式任务列表，支持基于文件锁的任务认领 (claim)。成员通过 team_task_create / team_task_claim 协调分工。

可选 Git Worktrees

每个成员可使用独立的 git worktree，防止并发编辑冲突。存储在 ~/.omo/teams/{name}/worktrees/ 下。

tmux 可视化

启用 tmux_visualization 后，每个成员在独立 tmux pane 中运行，Lead 可实时观察所有成员的工作状态。focus + grid 窗口布局。

成员资格注册表

eligible (Sisyphus, Atlas, Sisyphus-Junior)、conditional (Hephaestus 需显式允许)、hard-reject (Oracle, Librarian, Explore 等——用 task 代替)。

完整配置 Schema

// .opencode/oh-my-openagent.jsonc
{
  "team_mode": {
    "enabled": false,                    // 启用 Team Mode (需重启 OpenCode)
    "tmux_visualization": false,         // tmux 可视化布局
    "max_parallel_members": 4,           // 最大并行成员数 (1–8)
    "max_members": 8,                    // 团队总成员硬上限 (1–8)
    "max_messages_per_run": 10000,       // 每次运行的最大消息数
    "max_wall_clock_minutes": 120,       // 每次运行的最长执行时间
    "max_member_turns": 500,             // 每个成员的最大轮次

    "recipient_unread_max_bytes": 262144,// 接收方未读消息总上限 (≥1024)

}

Agent 排序机制

OmO 强制执行一个固定的、规范性的 Agent 排序：Sisyphus → Hephaestus → Prometheus → Atlas。这不是随意的惯例，而是影响 Agent 优先级和行为的关键保证。

实现机制

installAgentSortShim() 在插件初始化时通过补丁 Array.prototype.toSorted 和 Array.prototype.sort 实现排序。补丁仅当数组包含 ≥2 个规范核心 agent 时才激活，避免影响其他代码。

// 插件初始化——在 createPluginModule() 中
installAgentSortShim()
  // ↓ 补丁 Array.prototype.toSorted
  // ↓ 补丁 Array.prototype.sort (窄条件)
  // ↓ 仅当数组包含 ≥2 个核心 agent 时生效

排序规则

位置Agent原因
1Sisyphus主编排器，所有交互的入口
2Hephaestus主要深度工作者，第二高频调用
3Prometheus规划师，执行前调用
4AtlasTodo 编排器，计划执行阶段激活

此排序影响 OpenCode 中 Agent 列表的展示顺序、category 解析优先级以及某些内部路由决策。所有非核心 agent (Oracle, Librarian, Explore 等) 按字母顺序排在核心 agent 之后。

编排系统

OmO 的编排系统将 Agent 组织为三层架构，将意图转化为精确、可追踪的执行流水线。

三层架构

Agent 并非平等排列。它们按职责分为三个抽象层，每层独立优化：

层级包含 Agent职责
规划层 人类 + Prometheus + Metis + Momus 理解意图、分析需求范围、构建可执行计划、多角度审查验证。所有重大决策在执行前完成。
执行层 Atlas 将计划分解为 todo 项，按依赖顺序推进，协调 Worker 层，汇总执行结果向 Sisyphus 报告。
工作层 Sisyphus-Junior + Oracle + Explore + Librarian + category agents 具体执行原子任务：搜索代码、分析文档、编写代码、审查输出、运行验证。每个 worker 是独立 agent 实例。

这种分层设计的核心价值：每层只做一件事，并把那件事做好。规划层不写代码，工作层不做战略决策。关注点分离 (separation of concerns) 在 agent 系统层面得到了严格贯彻。

决策流程

Sisyphus 根据任务特征自动选择工作模式。决策树如下：

简单修复 → 直接执行

用户请求清晰、范围明确。Sisyphus 直接通过 prompt 响应，不经过额外规划流程。典型场景：修正变量名、修复文档 typo、调整导入顺序。

复杂 + 懒人模式 → ultrawork

任务复杂但不需要精确规格。Sisyphus 激活 Prometheus 做快速访谈，生成粗略计划后直接进入执行。适合探索性开发、功能原型、快速实验。

复杂 + 精确需求 → @plan 然后 /start-work

任务关键、范围模糊、需要严格遵守约束。先调用 @plan 启动完整三阶段规划 (Prometheus 访谈 → Metis 差距分析 → Momus 审查)，审查通过后 /start-work 激活 Atlas 加载计划逐项执行。

Prometheus 访谈模式

Prometheus 不比普通 Agent 更聪明——它写更好的提示词。通过像真正工程师一样迭代提问来缩小范围、解决歧义：

  1. 意图拆解——将模糊的用户请求转化为结构化问题列表。"你想要 A"→"B 场景下 A 的行为是什么？C 场景呢？"
  2. 约束发现——识别用户遗漏的边界条件、兼容性约束、性能要求。
  3. 方案推演——对关键决策点进行预验证："这个方案在 X 条件下成立吗？"
  4. 计划输出——生成 step-by-step 可执行计划，包含每步的完成标准和验证方法。

不同任务类型的访谈策略不同：

任务类型典型问题
重构/安全性关注现有行为保持、逆向兼容、测试覆盖率。第一个问题："你怎么知道改对了？"
从零搭建/探索关注技术选型、架构决策、扩展性。第一个问题："你确定这是正确的问题，而非症状？"
中型特性/防护栏关注边界条件、错误处理、用户预期。第一个问题："边界情况你考虑了吗？"
架构/战略关注权衡、长期影响、正交性。第一个问题："六个月后你还想要这个架构吗？"

Metis 与 Momus

规划阶段的两道质量防线，一前一后。

Metis 在计划最终确定前执行差距分析 (gap analysis)。它不写计划——只找问题。检查清单：

  • 隐藏意图——用户真正想要什么 vs 说了什么。有没有未表露的目标？
  • 歧义——哪些表述可被多种解释。需要澄清哪些点？
  • AI 失败模式——已知的 AI code smell 模式：幻觉、过时 API 假设、样板崇拜。
  • 缺失标准——验收条件是否完整，成功标准是否可测量？
  • 边缘情况——边界条件、空值、失败路径是否被考虑？

Momus 对已完成计划执行审查循环 (review loop)。四项黄金准则：

  • 清晰度 (Clarity)——每项任务是否精确描述了要做的事，而非意图？
  • 可验证性 (Verification)——完成标准是否可测量、可证明？不是"实现 X"而是"X 通过如下测试"？
  • 上下文完整性 (Context)——计划是否包含足够上下文，使执行者无需回头追问？
  • 大局观 (Big Picture)——是否有被遗忘的需求、依赖或约束？

Momus 的 OKAY 阈值非常严格：100% 文件引用有确切路径，≥80% 的决策有明确来源引用，≥90% 的验收条件可用具体方法证明，0 个未标记的假设。

Metis 和 Momus 都是只读 Agent。它们发现缺陷，但由人类或 Prometheus 决定是否修复。这是 OmO 规划哲学的核心部分——审查者不执行，执行者不审查。

Atlas 执行引擎

Atlas 不是普通的代码编写 Agent。它是指挥官 (conductor)，而非匠人 (craftsman)。

工作模式包含 6 个顺序阶段：

  1. 读计划——加载 Prometheus 阶段生成的 .omo/plans/*.md 计划文件，解析 todo 结构
  2. 分析——理解 todo 之间的依赖关系，确定最优执行顺序，识别可并行的任务
  3. 积累智慧——每次执行过程中记录 learnings，持续优化后续决策
  4. 委派——对复杂或探索性的子任务，通过 task() 委派给 Oracle 或 category agents
  5. 验证——每项 todo 完成后立即运行 lsp_diagnostics，确保零错误后再推进
  6. 报告——定期向 Sisyphus 汇报进度、阻塞项和阶段性结果

智慧积累系统 (Wisdom Accumulation)：Atlas 将运行过程中产生的洞察持久化到 .omo/notepads/{plan-name}/ 目录下：

文件用途
learnings.md新发现的代码库模式、必须遵守的约定、对项目的深层理解
decisions.md重大架构决策及其理由。避免同一问题反复讨论
issues.md遇到的 bug 和阻塞及其解决方式
verification.md验证步骤、测试结果、QA 证据链
problems.mdtodo 本身未覆盖的深层代码库问题

不要将 Atlas 与 Sisyphus 混淆：Sisyphus 是主编排器，面向用户交互；Atlas 是 todo 编排器，面向计划执行。Sisyphus 调用 Atlas 来推进计划，但只在高层次监控进度——具体执行细节完全由 Atlas 决定。

Sisyphus-Junior 约束

Sisyphus-Junior 是 category 路由的目标 Agent，设计上故意受限以保证可预测性：

  • 专注 (Focused)——不能委派任务、不能启动后台 task()。单一职责，只做被分配的那件事。
  • 纪律 (Disciplined)——强制 todo 痴迷。每步之前写 todo，完成后立即标记完成。不允许跳过验证。
  • 验证 (Verified)——每项修改后必须运行 lsp_diagnostics，确保零错误。不干净不前进。
  • 受限 (Constrained)——只能读取计划文件 (read-only)。不能修改 config、schema、package.json 等关键文件。

为什么回退链足够？Sisyphus-Junior 承载的任务相对简单——通过 category 路由的通常是特定领域的原子任务。系统的智能 (编排、规划、审查) 由顶层 Agent 完成，worker 层不需要最强模型。这是 OmO 的核心设计权衡：系统智能 > worker 模型能力。编排和审查的质量比执行模型的能力更影响最终结果。

System Reminder 机制：Boulder 状态系统在 todo 长时间无进展时，通过 <system-reminder> 推送提醒，确保 Sisyphus-Junior 不会偏移任务目标。这是系统的"执拗性"保障。

Team Mode 详解

Team Mode 将 OmO 从"一个 Agent + 若干子 Agent"的架构，升级为真正的多 Agent 协作系统。一个 Lead Agent 协调一个并行运行的专家团队，通过专用通信和协调工具协作。

何时使用

Team Mode 不是默认开启的——它有显著的开销 (上下文窗口、通信延迟)。最适合以下场景：

  • 并行探索——需要同时从多个方向调研。例如同时研究前端框架选型和后端数据库方案，两个成员并行，时间减半。
  • 长时间重构——大型代码库拆分，多个成员独立处理不同模块，每个成员在自己的 worktree 中工作。
  • 研究 + 实现流水线——部分成员负责研究可行性方案和 API 设计，另一部分成员同步开始原型实现。

配置与启用

在项目配置文件 .opencode/oh-my-openagent.jsonc 中启用：

{
  "team_mode": {
    "enabled": true,
    "max_parallel_members": 4,
    "tmux_visualization": false
  }
}

已知问题：v4.2.1 之前的版本存在配置读取 bug——启用 Team Mode 后有时需要重启 OpenCode 两次才能使配置完全生效。此问题已在 v4.2.1 中修复。如果你使用的是旧版本，可在 Team Mode 技能 (hyperplan, security-research) 的配置中单独设置 "team_mode": {...} 块作为临时方案。

成员资格

不是所有 Agent 都能加入团队。AGENT_ELIGIBILITY_REGISTRY 定义了三条严格的资格规则：

资格Agent说明
eligible (合格) Sisyphus, Atlas, Sisyphus-Junior 默认允许加入团队。可直接通过 kind: "subagent_type" 声明为成员。
conditional (条件合格) Hephaestus 默认不具备 teammate: "allow" 权限。需在 tool-config-handler.ts 中显式授予 D-36 权限，或使用 subagent_type: "sisyphus" 代替。
hard-reject (硬拒绝) Oracle, Librarian, Explore, Multimodal-Looker, Metis, Momus, Prometheus 解析阶段直接拒绝。它们的设计目的与团队协作不兼容——要么只读、要么规划、要么审查。用 task() 委派代替。

为什么 Oracle 和 Prometheus 被硬拒绝？Oracle 是只读顾问——它不写代码，不编辑文件。在需要读写的团队协作中，Oracle 的角色被 Lead 的编排和 task() 调用替代。Prometheus 是规划师——规划完成后它的使命就结束了。在团队场景中，Lead 本身承担了规划职责，Prometheus 的访谈模式更适合在团队启动前使用。

生命周期

一个团队从创建到销毁经历 5 个阶段：

  1. 创建 (team_create)——Lead 调用 team_create 工具定义成员列表、配置文件、初始化共享状态 (mailbox、tasklist)。
  2. 委派 (Delegate)——Lead 通过 team_send_message 向成员分配任务。成员从 mailbox 轮询获取新消息 (mailbox_poll_interval_ms 控制轮询频率)。
  3. 认领与完成 (Claim & Complete)——成员通过 team_task_claim 从共享 tasklist 中认领任务，执行完毕后 team_task_update 更新状态和结果。
  4. 关闭 (Shutdown)——Lead 确认所有任务完成，调用 team_shutdown_request。各成员通过 team_approve_shutdown 确认或 team_reject_shutdown 拒绝 (如有未完成任务)。
  5. 删除 (Delete)——所有成员确认关闭后，清理 ~/.omo/teams/{name}/ 下的所有工件。

Worktrees

Team Mode 支持为每个成员分配独立的 git worktree，彻底消除并发编辑冲突。存储在 ~/.omo/teams/{name}/worktrees/{memberName}/ 下。

工作流程：成员启动时自动创建 worktree → 成员在独立分支上工作 → 成员完成时自动提交变更 → Lead 从各 worktree 拉取变更 → 合并到主分支。冲突发生时，Lead 负责解决。

Worktrees 不是默认启用的——仅在成员数 > 1 且有文件级并发风险时推荐启用。

TMUX 可视化

启用 tmux_visualization: true 后，团队的每个成员在独立的 tmux pane 中运行。布局设计：

  • Lead pane 位于焦点位置 (主窗口左/上侧)
  • 成员 pane 以 grid 布局排列 (右/下侧)，每个 pane 实时流式输出
  • Lead 可随时扫视任意 pane 了解成员当前状态
  • Team Mode 结束时自动恢复原始 tmux 布局

这解决了多 Agent 系统最头疼的问题之一——可观察性。没有 tmux 可视化时，Lead 只能通过 mailbox 消息间接了解成员进度，延迟高且信息碎片化。tmux 布局将不可见变为可见。

限制

Team Mode 并非万能。以下是明确的不适用范围：

  • 不支持嵌套团队——团队成员不能创建子团队。展开子问题用 task() 委派给 category agent 即可。
  • 不支持同步回复——所有团队通信基于 mailbox 轮询。通信延迟由 mailbox_poll_interval_ms (默认 3s) 决定。
  • 成员不能调用 delegate-task——只有 Lead 拥有委派权限。成员专注于分配的任务，不能自行扩散工作范围。
  • 无单方面强制关闭——关闭流程需要 Lead 发起 + 所有成员确认。任何成员可拒绝关闭。

使用模式

理解 Agent 间不同的协作模式，在不同场景中选择最合适的调用方式。

如何调用 Prometheus

两种方式启动 Prometheus 的访谈式规划：

方法操作行为适用场景
方法 1: Switch Agent 通过 Agent 选择界面主动切换到 Prometheus Prometheus 完全接管对话，直接开始访谈流程 明确知道需要深度规划、任务模糊度极高时
方法 2: @plan 命令 在消息中嵌入 @plan 指令 Sisyphus 自动调用 Prometheus 进行规划，完成后将计划转交 Momus 审查，然后返回可执行计划 日常开发中顺带触发规划，不想手动切换 Agent 时

/start-work 与会话连续性

/start-work 是连接规划阶段和执行阶段的关键桥梁。它向 Atlas 发出信号，加载 Prometheus 生成的可执行计划并开始执行。

背后机制：boulder.json

Prometheus 生成计划后，系统在 .omo/plans/{plan-name}/ 下创建 boulder.json。该文件记录：所有 todo 项、每项的完成状态、依赖关系、验证标准。Atlas 读取此文件来恢复和推进工作进度。

两种启动模式：

  • 初始模式 (Init)——首次启动新计划。Atlas 读取 plan.md，解析 todo 结构，写入 boulder.json，从第一项开始执行。
  • 恢复模式 (Resume)——会话中断后重新启动。Atlas 读取 boulder.json 的已有进度，标记已完成的 todo，从第一个未完成项继续。

典型执行时间线：

  1. 用户提出复杂需求 → Sisyphus 判断需要正式规划
  2. 用户输入 @plan 实现用户认证系统 → Prometheus 启动访谈
  3. Prometheus 产出 .omo/plans/auth-system/plan.md → Momus 审查
  4. 用户输入 /start-work → Atlas 加载计划 → 写入 boulder.json
  5. Atlas 逐项执行 todo → 每项完成后 lsp_diagnostics 验证 → 更新 boulder.json
  6. 所有 todo 完成 → Atlas 向 Sisyphus 报告 → Sisyphus 汇总最终结果给用户
  7. 中断恢复：新会话中 /start-work → Atlas 读取 boulder.json → 从断点继续

Hephaestus vs Sisyphus + ultrawork

OmO 提供两种"深度工作"模式。它们不是竞争关系，而是针对不同任务特征优化的互补方案：

维度HephaestusSisyphus + ultrawork
推荐模型GPT-5.5Claude Opus 4.7 / Kimi K2.6
工作方式自主探索 + 端到端执行。给目标，不给步骤。Todo 驱动 + 委派。Sisyphus 规划并协调多个专家 agent。
最佳场景探索性开发、端到端特性实现、代码库范围重构需要精确步骤的复杂任务、多文件多模块变更、严格质量把关
规划策略内隐式——Hephaestus 自主决定下一步做什么，不做显式记录外显式——Prometheus 生成书面计划，Momus 审查，Atlas 按计划执行
委派倾向Hephaestus 可以调用 task()，但倾向于自主完成Sisyphus 频繁委派，最大化并行度，让专家做专家的事
温度稍高——鼓励创造性探索和非常规解法较低——强调纪律性、可追溯性和输出一致性
可追溯性低——Hephaestus 不记录决策过程，结果导向高——每一步都有 plan、review、boulder.json 记录

选择原则：需要创造力 + 自主性 → Hephaestus。需要可靠性 + 可追溯性 → Sisyphus + ultrawork。如果你不确定——默认用 ultrawork。Hephaestus 适合你明确知道"做什么"但不确定"怎么做"的场景。

Category + Skill 系统

Category 是 OmO 最具颠覆性的抽象之一。它从根本上解耦了"做什么类型的工作"和"用什么模型来做"。

为什么 Category 是革命性的？

传统系统要求用户或 Agent 手动选择模型。但模型名称本身会带来分布偏差 (distributional bias)——如果 Agent 的 system prompt 里写着"你运行在 Claude Opus 上"，即使 prompt 的其余部分完全一致，其行为也会与"你运行在 GPT-5.4 Mini 上"不同。Agent 会不自觉地"演"出它以为该模型应有的风格。

Category 打破了这种耦合。Agent 不需要知道自己在什么模型上运行。它只需要知道它做的是 "quick" 还是 "ultrabrain" 类的工作。系统在背后完成任务到模型的匹配。

// Category 路由——模型完全自动决定
task(
  category="quick",                  // 自动路由到 GPT-5.4 Mini
  description="Fix typo in readme",
  prompt="Fix the typo in line 23 of README.md"
)

// Skill + Category 组合——加载领域知识
task(
  category="ultrabrain",
  load_skills=["security-research"],  // 加载安全审计 skill
  description="Audit payment flow",
  prompt="Find vulnerabilities in payment processing..."
)

// Skill 加载优先级 (由近到远)
// 1. project — .opencode/skills/*/SKILL.md
// 2. opencode — 系统自带 skills
// 3. user — ~/.config/opencode/skills/*/SKILL.md
// 4. builtin — 插件内置 skills

Skill MCP 隔离：每个 skill 的 MCP 服务器按 ${sessionID}:${skillName}:${serverName} 键隔离。同一 skill 在两个不同会话中启动的 MCP 实例不共享任何状态。这从根本上防止了上下文污染和认证凭据冲突。

Background Task 并发

Background Agent 系统提供生产级别的并发控制：

  • 默认并发：每个 providerID/modelID 组合最多 5 个并发后台任务
  • 可配置：通过 background_task.modelConcurrency 和 background_task.providerConcurrency 分别调整模型级别和提供商级别的并发上限
  • FIFO 队列：超出并发上限的任务进入 FIFO 队列，等待空闲槽位自动启动
  • 上下文隔离：每个后台任务拥有独立的会话上下文，不污染主 session 的上下文窗口
  • 完成通知：后台任务完成后，系统通过 <system-reminder> 通知主 session，附带 task_id。主 session 通过 background_output(task_id) 取回完整结果

禁止使用 background_cancel(all=true)：全量取消会导致不确定状态。始终通过 taskId 逐个取消后台任务。这是系统的硬性约束——如果不遵守，未完成的任务可能在取消后继续写入 session 状态，造成数据损坏。
