第三层 · Hook 与工具系统

Hook 是 OmO 的生命线——在每个关键生命周期点拦截、变换、守卫、延续会话。工具是 Agent 的手和眼——从文件读写到 MCP 搜索，从后台任务到 Team 协作。

五层 Hook 组合架构

Hook 系统采用 5 层组合模式，每层负责不同关注面：

Session 层 — 23 个 Hook
会话级事件处理：keyword-detector / runtime-fallback / boulder-state / 生命周期事件分发
ToolGuard 层 — 17 个 Hook
工具执行前守卫：write-existing-file-guard / rules-injector / prometheus-md-only / label-truncator
Transform 层 — 4 个 Hook
消息 / 提示变换：context-injection / thinking-block-validation
Continuation 层 — 7 个 Hook
会话持续机制：todo-continuation / compaction-autocontinue
Skill 层 — 2 个 Hook
技能生命周期：skill-mcp-manager
+ Team Mode — 7 个 Hook (条件)
team_mode.enabled 时注入：team-tool-gating (ToolGuard +1) / team-mode-status-injector (Transform +1) / team-mailbox-injector (Transform +1) / 4 个直接事件处理器 (team-session-events)
层级数量关注面典型 Hook

Hook 系统通过 createCoreHooks() + createContinuationHooks() + createSkillHooks() 组合而成。Team Mode 的 7 个额外 hook 在 createHooks() 中条件注入。每个 hook 可通过 disabled_hooks 配置项单独禁用。

Hook 实例解析

Ultimate

keyword-detector

IntentGate 关键词检测器。在会话首条消息触发，分类用户意图：ultrawork / search / analyze / team。根据分类注入对应的模式专用系统提示，确保 Agent 在正确模式下运行。位于 hooks/keyword-detector/。

Ultimate

runtime-fallback

当模型调用失败时触发。包含 first-prompt-watchdog：子 agent 90 秒无进度 → 触发回退或中止。区分两套独立回退系统：model-fallback (主动，在 chat.params 中) 和 runtime-fallback (被动，在 session.error 中)，二者相互独立。

Ultimate

hashline-read-enhancer

每个 Read 工具的输出附加 LINE#ID 内容哈希标签。哈希字符来自 ZPMQVRWSNKTXJBYH。编辑时通过哈希验证行内容是否变更——过时哈希直接拒绝编辑，消除因文件在读取后被修改而造成的错行错误。

Both

comment-checker

检测 AI 生成代码异味的模式匹配器。识别 "simply" "obviously" "clearly" 等 AI 写作特征。单行绕过：// @allow；文件级绕过：// comment-checker-disable-file。打包为独立二进制 @code-yeongyu/comment-checker。

Both

rules-injector

在每次 prompt 中自动注入项目规则。扫描 AGENTS.md、.omo/rules/*.md、.claude/rules/、.cursor/rules/、.github/instructions/、.github/copilot-instructions.md 和 .mdc 文件。采用向上遍历发现机制。

Ultimate

prometheus-md-only

Prometheus Agent 只允许编辑 .md 文件。在 tool.execute.before 守卫中强制检查：任何非 .md 文件的编辑操作被拒绝。禁止路径：packages/*/src/、package.json、配置文件。

Ultimate

team-tool-gating

Team Mode 启用时注入的 ToolGuard hook。对 team_* 系列工具执行权限守卫——验证当前会话是否有权创建 / 管理 team、成员是否超出限制、配置是否符合安全边界。

三级 MCP 系统

OmO 的 MCP 系统分三层，从内置到兼容到技能嵌入，逐级扩展：

层级来源加载器机制
1. 内置 MCP createBuiltinMcps()
3 远程 HTTP: websearch (Exa) / context7 (docs) / grep_app (GitHub 搜索)
+ 2 本地 stdio: lsp (packages/lsp-tools-mcp) / ast_grep (packages/ast-grep-mcp)		 插件初始化时自动创建 运行时常驻，所有会话共享
2. Claude Code 兼容 .mcp.json (项目级 + 用户级) claude-code-mcp-loader ${VAR} 环境变量展开，受 mcp_env_allowlist 白名单控制
3. Skill 嵌入式 SKILL.md YAML frontmatter SkillMcpManager (按会话) stdio + HTTP 传输，OAuth 2.0 + PKCE + DCR step-up，按需启动按需销毁

按会话 MCP 隔离

第三层 MCP 客户端按 ${sessionID}:${skillName}:${serverName} 键隔离——同一 skill 在不同会话中不共享 MCP 状态。技能结束时 MCP 服务器自动销毁。

设计原则：MCP 服务器消耗上下文预算。Skill 嵌入式 MCP 按需启动、任务作用域限制、完成后销毁——上下文窗口保持干净。

6 阶段配置管道

插件初始化时，配置按 6 阶段依次加载。每个阶段的结果传递给下一阶段：

阶段 1: provider
模型供应商配置 (Anthropic / OpenAI / Gemini / Copilot 等)
阶段 2: plugin-components
插件核心组件注册 (features, skills, commands)
阶段 3: agents
Agent 定义、模型绑定、工具权限
阶段 4: tools
工具注册表构建 (含条件开关评估)
阶段 5: MCPs
三级 MCP 系统组装
阶段 6: commands
内置 command / slash 命令注册

多级配置合并规则

遍历配置 (越近优先级越高):
  <从 pwd 到 $HOME>/.opencode/oh-my-openagent.json[c]
        ↓ 合并到
  用户配置: ~/.config/opencode/oh-my-openagent.json[c]
        ↓ 回退到
  默认值 (Zod safeParse 填充省略字段)
字段类型合并策略说明
agents, categories, claude_code 深合并 防护原型污染。内层嵌套对象递归合并。
disabled_* 数组 Set 合并 拼接后去重。禁用列表只增不减。
其他所有字段 覆盖替换 较近的配置覆盖较远的。基本类型全覆盖。
mcp_env_allowlist 仅限用户配置 安全原因——遍历配置不可扩展环境变量白名单。

配置格式：JSONC (支持注释和尾逗号)，使用 Zod v4 验证，字段名遵循 snake_case。迁移通过 migrateConfigFile() 处理——基于 _migrations 追踪的幂等迁移 + 时间戳备份。
