架构深度解析

38 个 monorepo 包按严格层级组织。依赖规则：DAG 只能向下流动。上层依赖下层，下层不知道上层存在。

分层总览

Web — 营销网站
packages/web/ — Next.js 15 + Cloudflare Workers，独立应用，不使用路径别名外的 monorepo 包。自有 bun.lock 和 tsconfig。
Platform — 平台二进制
11 平台二进制 — bun compile 编译产物 (darwin/linux/windows × arch × variant)。部署产物，不被任何包导入。包名模式：oh-my-opencode-<os>-<arch>[-variant]。由 publish-platform.yml 生成。
Adapters — 适配器层 (2 包)
omo-opencode/ + omo-codex/ — 针对各 harness 的薄包装。导入 Core，包装成 harness API，导出。这是唯一知道 OpenCode 或 Codex 具体 API 的层。
Skills — 技能声明 (1 包)
shared-skills/ — SKILL.md 静态声明文件。Markdown 被 agent 消费，不含运行时代码。跨 harness (OpenCode + Codex) 共享。
MCP — 工具服务器 (4 包)
lsp-tools-mcp, ast-grep-mcp, git-bash-mcp, lsp-daemon — stdio 进程边界隔离。host-agnostic，不依赖任何 harness。LSP 包消费 lsp-core + mcp-stdio-core。
Core — 纯逻辑核心 (19 包)
纯 TypeScript 逻辑。无 harness 依赖。可隔离测试。所有层的基础。

依赖方向： Web 和 Platform 是叶子节点。Adapter 依赖 Core、MCP、Skills。没有包依赖 Adapter。MCP 依赖 Core。Skills 不依赖任何代码包。Core 是最底层，不依赖任何其他包。

Core 层：19 个纯逻辑包

每个 Core 包都是纯 TypeScript，无 harness 依赖，可隔离测试。适配器通过 workspace 依赖消费这些包。

包名 (packages/)用途
utils 深层合并、snake_case 转换、frontmatter 解析等跨包通用工具。
model-core 模型解析管线：provider 缓存 (ProviderCache DI)、模型能力匹配、模型需求解析。
prompts-core harness-neutral 的 markdown prompt 加载器。模式路由 (ultrawork/search/analyze/team)，不依赖任何 harness。
rules-engine 规则发现与匹配引擎。扫描 .omo/rules/、.claude/rules/ 等目录，按条件注入规则。原名 rules-core。
agents-md-core AGENTS.md walk-up 发现与上下文注入。从工作目录向上直至 $HOME，收集并聚合 AGENTS.md 内容。
ast-grep-core ast-grep 类型定义、模式提示、runner 核心。不包含 CLI 或 MCP 包装。
comment-checker-core apply-patch 解析器与二进制 runner。检测 AI slop 注释模式并拒绝。
hashline-core Hashline 编辑原语：LINE#ID 内容哈希生成、diff 助手、哈希验证。灵感来自 oh-my-pi。
boulder-state 工作追踪状态机：boulder 创建/更新/完成，分体存储 (boulder-state 数据 + metadata 元数据)。
telemetry-core harness-neutral 的遥测：PostHog 包装器，DAU/WAU/MAU 追踪，SHA256 hashed 安装 ID。
lsp-core harness-neutral 的 LSP 引擎：请求上下文、工具定义、语言服务器管理。被 lsp-tools-mcp 和 lsp-daemon 消费。
mcp-stdio-core JSON-RPC stdio 帧协议：dispatch 原语、消息序列化、进程生命周期管理。
tmux-core harness-neutral 的 tmux 操作：会话/窗格/布局/runner。Team Mode 可视化依赖此包。
claude-code-compat-core Claude Code 兼容加载器：plugins、MCPs、commands、agents 的兼容适配，不含 harness 特定胶水。
skills-loader-core Skill 加载：builtin skill、runtime skill、skill 匹配与发现。
mcp-client-core MCP 客户端生命周期：skill-embedded MCP 管理器、OAuth 2.0 + PKCE + DCR 流程。
openclaw-core OpenClaw 网关：reply-listener 守护进程、会话注册表、tmux 注入。
team-core Team Mode 核心：registry、mailbox、tasklist、state machine、worktree、tmux 布局。不包含 OpenCode 工具注册。
delegate-core 委派任务选择：category 匹配、retry 原语、任务优先级排序。

初始化流程：14 阶段管道

createPluginModule() 按顺序执行 14 个阶段，每个阶段都可能失败并提前退出。下面是完整流程：

#阶段说明
1 installAgentSortShim() 修补 Array.prototype.toSorted/.sort，当数组包含 ≥2 个核心 agent 时强制规范排序 (Sisyphus→Hephaestus→Prometheus→Atlas)。
2 initConfigContext() 检测 opencode-vs-openagent 布局标志，处理包名迁移兼容。
3 logLegacyPluginStartupWarning() 如果通过旧条目 oh-my-opencode 加载，记录弃用警告。
4 migrateLegacyWorkspaceDirectory() 首次加载时将 .sisyphus/ 状态复制到 .omo/。
5 detectDuplicateOmoPlugin() 检测到重复的 omo/openagent 插件时提前退出，防止双重加载。
6 detectExternalSkillPlugin() 检测冲突的外部 skill 插件并发出警告。
7 injectServerAuthIntoClient() 将认证头注入共享 SDK 客户端，使内部请求可携带服务器端 auth。
8 loadPluginConfig() 6 阶段配置管道：JSONC 解析 → 用户/项目配置合并 → Zod v4 验证 → 迁移旧键。越近的配置优先级越高。
9 selectRuntimeSecuritySkills() + createRuntimeSkillSourceServer() 选择运行时安全技能，创建按需 skill source server。
10 initI18n() 加载本地化字符串 (packages/omo-opencode/src/locales/)。
11 setAgentSortOrder() 应用用户配置的 agent_order，覆盖默认排序。
12 initializeOpenClaw() 如果存在 openclaw 配置：启动双向 Discord/Telegram/HTTP/shell 网关。
13 checkTeamModeDependencies() 如果 team_mode.enabled：检查 tmux 等依赖。失败则降级为 disabled-skills 警告。
14 startTmuxCheck() → ... 最后的批量创建：tmux 检查 → createManagers() → createTools() → createHooks() → createPluginInterface() → createPluginDispose()。全部完成后插件就绪。

重构状态

当前阶段：Package Layering Refactor — 将 packages/ 重构为严格分层 (Core → MCP → Skills → Adapters → Platform → Web)，使同一套逻辑可跨 harness 重用，消除三个仓库 (omo、pi-extensions、codex-plugins) 之间的代码重复。

已完成

  • 19 个 Core 包已从原位置提取到 packages/ 下
  • omo (OpenCode 适配器) 通过 workspace 依赖消费这些 Core 包，适配器 shim 留在原 packages/omo-opencode/src/ 位置作为稳定锚点
  • lsp-tools-mcp 和 lsp-daemon 已迁移为消费 lsp-core + mcp-stdio-core
  • 分层已达成：Core (19 纯 TS 包) → MCP 包 → Adapters → Platform 二进制

下一阶段

  • pi-extensions 和 codex-plugins 尚未迁移到消费 Core 层。该迁移是下一阶段。

未来方向 (探索性)

  • 通过共享 Core 层支持多个 harness：Claude Code、Codex、Pi、Amp、Droid 等。
  • Pi Engine DI 抽象已推迟。可在适配器迁移完成后重新评估。
  • 不追求"大一统"插件接口，不追求人类可读的目录结构优先于 agent 循环性能。
原则：当有疑问时，选择对执行工作的 agent 需要最少推理的表示形式。如果这让目录对人类显得混乱，那么这个目录对人类是错的，对 agent 是对的。

架构不变量

以下规则不得违反。它们被测试和 lint 规则强制，违反会导致构建失败或运行时错误。

不变量规则实施
规范 agent 排序 Sisyphus → Hephaestus → Prometheus → Atlas。当数组包含 ≥2 个核心 agent 时强制此顺序。 installAgentSortShim() 修补 Array.prototype。order 字段注入。
Hashline edit+read 配对 每次 Read 工具输出必须带 LINE#ID 内容哈希。edit 在应用前验证哈希。哈希不匹配 → 拒绝。 hashline edit 工具在写入前检查。stale hash 直接拒绝。
Prompt-async gate session.prompt / session.promptAsync 是危险的 — OpenCode 在 prompt 被持久接受前就返回。所有内部消息注入必须通过 prompt-async-gate.ts。 静态审计测试 (prompt-async-route-audit.test.ts) 扫描整个代码库，发现 gate 外的原始 prompt 调用则失败。
OpenClaw 双向 出站 dispatcher 在会话事件时触发。入站守护进程轮询 Discord/Telegram，并将回复 send-keys 到被追踪的 tmux 窗格。 openclaw-core 会话注册表 + tmux 窗格追踪。
按会话 MCP 隔离 第三级 MCP 客户端以 ${sessionID}:${skillName}:${serverName} 为键。同一 skill 在两个会话中不共享状态。 per-session MCP manager keying。
两个 fallback 系统独立 model-fallback (主动，chat.params，硬编码链) 与 runtime-fallback (响应式，session.error，按 category/agent 配置) 独立运行，互不集成。 两个模块无交叉导入。独立触发路径。

PR 合并策略

必须使用 merge commit。命令：gh pr merge <num> --merge --delete-branch。绝不使用 squash-merge 或 rebase-merge，即使通用工作流、skill 或 GitHub 默认建议如此。此仓库级规则覆盖所有其他建议。

规则说明
目标分支所有 PR 必须 target dev。target master 的 PR 被硬阻止。
合并方式--merge 仅。创建 merge commit，不 squash，不 rebase。
合并前CI 通过 (tests + typecheck + build)，review-work 审查通过，Cubic 通过 (除非配额耗尽)。
工作流使用 work-with-pr skill：隔离的 git worktree → 实现 + QA → PR → 验证循环 → merge。

CI/CD 工作流

工作流触发用途
ci.yml push/PR 到 master/dev 测试、类型检查、构建、codex 兼容性 (ubuntu/macos/windows)。master push 时自动提交 schema。dev push 时创建 draft "next" release。
publish.yml 手动 dispatch 双 npm publish (oh-my-opencode + oh-my-openagent) + lazycodex-ai npm 包 + 自动 Codex marketplace 同步 + 平台二进制 + GitHub release + merge 到 master。
publish-platform.yml 由 publish.yml 调用 通过 bun compile 构建 11 平台二进制 (darwin/linux/windows)。
sisyphus-agent.yml @mention 或手动 dispatch AI agent 处理 issues 和 PRs。
refresh-model-capabilities.yml 每周 cron / dispatch 从 models.dev API 刷新模型能力缓存。
cla.yml issue_comment / PR CLA 助手，管理 contributor license agreement 签名。
lint-workflows.yml push/PR 涉及 .github/workflows/** actionlint 检查，shellcheck 已禁用。
web-ci.yml push/PR 涉及 packages/web/**, docs/** 格式检查、lint、类型检查、next build、opennextjs-cloudflare build。另有 web-deploy.yml 部署到 Cloudflare Workers。

关键约定

类别约定
运行时 仅 Bun (CI 中 1.3.12+)。永不使用 npm/yarn/pnpm 运行项目。例外：lsp-tools-mcp 和 lsp-daemon 是 Node-targeted，使用 npm + vitest/biome。
TypeScript strict mode、ESNext、bundler moduleResolution、bun-types。永不使用 @types/node。永不使用 as any、@ts-ignore、@ts-expect-error。
测试 Bun test (bun:test)，与源文件并列的 *.test.ts。given/when/then 风格 — 嵌套 describe 使用 #given/#when/#then 前缀，或行内注释。永不使用 Arrange-Act-Assert。
工厂模式 所有工具、hook、agent 使用 createXXX() 工厂函数。
文件命名 所有文件和目录使用 kebab-case。
模块结构 index.ts barrel 导出。禁止 catch-all 文件 (utils.ts、helpers.ts、service.ts)。每文件 200 行软限制。
导入 模块内使用相对导入，跨模块使用 barrel 导入。packages/src/ 内部禁止路径别名 — 永不使用 @/。例外：packages/web/ 使用 @/* (Next.js 约定)。
配置格式 JSONC (注释 + 尾逗号)、Zod v4 验证、snake_case 键名。
空 catch 禁止空 catch 块 catch(e) {}。
日志 写入 OS 临时目录下的 oh-my-opencode.log。50 MB 轮转，保留 .1 和 .2 备份。
后台任务 每个 ${providerID}/${modelID} 键 5 并发 (可配置)。队列满时 FIFO。
