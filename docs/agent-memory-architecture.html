01Memory 是什么？做什么？

从零开始理解 agent memory 的基本概念

想象你和一个人聊天，每次对话结束后，你希望系统能"记住"关于这个人的信息：他的偏好、他的技能使用习惯、发生过的重要事件等。

Agent Memory 就是一套自动从对话中提取并持久化这些"记忆"的系统。

在这个项目中，memory 的实现位于 session/memory/ 目录下。整个流程可以简化为：

核心流程 对话消息  →  LLM 提取  →  持久化到文件系统  →  后续对话可检索使用

它不是简单的"把聊天记录存下来"，而是让 LLM 理解对话内容，然后结构化地提取出有关用户和 Agent 的知识，分别存入不同类型的记忆文件中。

02核心概念：Memory Type（记忆类型）

Memory 不是"大锅烩"，而是按类型精确分类

每种记忆类型由 YAML 配置定义，位于 prompts/templates/memory/ 目录。当前定义了 12 种记忆类型：

类型用途存储路径
profile用户身份摘要（年龄、职业等）viking://user/{user}/memories/profile.md
identity用户身份声明卡片viking://user/{user}/memories/identity/
events真实世界发生的事件viking://user/{user}/memories/events/{年}/{月}/{日}/
skills技能使用统计与经验viking://agent/{agent}/memories/skills/
tools工具使用统计与经验viking://agent/{agent}/memories/tools/
soulAI 的"灵魂"设定viking://agent/{agent}/memories/soul.md
preferences用户偏好（习惯、品味）viking://user/{user}/memories/preferences/
entities实体（人、地点、事物）viking://user/{user}/memories/entities/
experiencesAgent 经验知识viking://agent/{agent}/memories/experiences/
trajectoriesAgent 行为轨迹viking://agent/{agent}/memories/trajectories/

每种类型通过 YAML 描述其 fields（字段）、merge_op（合并策略）、content_template（文件模板）等。以 skills.yaml 为例：

YAMLmemory_type: skills
directory: "viking://agent/{{ agent_space }}/memories/skills"
filename_template: "{{ skill_name }}.md"
fields:
  - name: skill_name
    type: string
    merge_op: immutable      # 一旦设定不可修改
  - name: total_executions
    type: int64
    merge_op: sum            # 每次累加
  - name: best_for
    type: string
    merge_op: patch          # 使用 SEARCH/REPLACE 局部修改

这种声明式设计意味着：新增记忆类型只需写一个 YAML 文件，零代码改动。

03数据模型层

核心数据结构的类型体系

在 dataclass.py 中定义了整套类型系统，它们是整个 memory 系统的"语言"：

MemoryTypeSchema → 描述一种记忆类型（从 YAML 解析而来） ├── MemoryField → 描述一个字段（名称、类型、合并策略、初始值） MemoryData → 运行时的一条记忆数据（动态字段容器） MemoryFile → 已持久化记忆文件的解析结果（content + links + backlinks） ResolvedOperation → 一条解析后的操作（增/改），含目标 URI 和字段数据 ResolvedOperations → 一次提取的所有操作集合（增/改/删） WikiLink → LLM 输出的临时链接（使用 page_id） StoredLink → 持久化后的链接（使用 URI）

PageIdMap

在单次提取生命周期内，维护 page_id（临时数字 ID）到 URI（文件路径）的映射。已有页面分配 1-99，新页面分配 100+。这让 LLM 可以用简单数字创建链接关系。

FaultTolerantBaseModel

容错基类。在 Pydantic 验证前自动处理 LLM 输出的不标准格式 —— 如 "None" 字符串转 None、数组转字符串、非法枚举回退到默认值等。

04Merge 操作机制

Memory 不是简单的"覆盖写入"，而是四种精密的合并策略

这是整个系统设计中最精妙的部分之一。定义在 merge_op/ 目录：

MergeOp含义LLM 看到的 Schema应用场景
patch 增量修改（默认） StrPatch 结构（SEARCH/REPLACE 块） 内容丰富的字段，如技能指南
replace 完全替换 纯字符串 需要整体重写的字段
sum 数值累加 整数/浮点数 计数类字段，如执行次数
immutable 不可变 基础类型 标识类字段，如技能名称
Patch 是最巧妙的设计 例如对一个已存在的 skills.md，LLM 输出的是： 
{
  "best_for": {
    "blocks": [
      {"search": "old description text", "replace": "new description text"}
    ]
  }
}
系统用 search 文本在原文件中定位，替换为 replace 文本。这避免了让 LLM 重写整个字段，节省 token 并减少出错概率。

MergeOpFactory（merge_op/factory.py）是工厂类，从 MemoryField 的定义创建对应的 MergeOp 实例。

Link Merge（链接合并）

另一个特殊合并：处理记忆页面之间的链接关系。去重 key 是 from_uri + to_uri + match_text，权重冲突取 最大值，类型和描述以后写入的为准。

05架构核心：ReAct 循环

整个 memory 系统的心脏 —— ExtractLoop

ExtractLoop（extract_loop.py）实现了经典的 ReAct（Reasoning + Acting）模式：

┌─────────────────────────────────────────────┐ │ ExtractLoop.run() │ │ │ │ ① 获取 schemas（记忆类型定义） │ │ ② SchemaModelGenerator 动态生成 Pydantic 模型 │ │ ③ 构建 System Prompt（含 JSON Schema） │ │ ④ Pre-fetch：自动搜索/读取已有记忆 │ │ │ │ ┌──── ReAct 循环 (max 3轮) ────────┐ │ │ │ │ │ │ │ LLM 调用 │ │ │ │ ├── 返回工具调用？→ 执行并继续 │ │ │ │ └── 返回操作 JSON？→ 解析 │ │ │ │ │ │ │ └──────────────────────────────────────────┘ │ │ │ │ ⑤ 解析操作 → ResolvedOperations │ │ ⑥ 交给 MemoryUpdater 持久化 │ └─────────────────────────────────────────────┘

SchemaModelGenerator

在运行时根据 YAML schema 动态创建 Pydantic 模型。每个 memory_type 生成一个 data 模型，最终组合成 StructuredMemoryOperations —— 这就是 LLM 需要输出的 JSON schema。

Pre-fetch 阶段

在 LLM 调用之前，系统自动执行语义搜索，找到相关已有记忆并注入到上下文中。让 LLM 知道"现在已有的记忆是什么"，避免重复或冲突。

LLM 可以在循环中使用 read 和 search 工具进一步探索已有记忆，也可以在信息足够时直接输出操作 JSON。处理完毕后通过 resolve_operations() 将 page_id 解析为真实 URI，通过 finalize_operations() 将 WikiLink 解析为 StoredLink。

06上下文提供者（Context Provider）

为 LLM 提供"看到什么"的桥梁

SessionExtractContextProvider 是 ExtractContextProvider 接口的主要实现。它负责：

  • 提供指令（instruction()）：告诉 LLM 它是 memory extraction agent，只能读不能写
  • Pre-fetch（prefetch()）：将对话历史格式化、对所有记忆目录执行语义搜索、读取单文件 schema
  • 工具执行（execute_tool()）：LLM 调用的 read/search 工具通过这里执行
  • 提供 schemas：过滤掉 agent_only 的记忆类型（trajectory/experience 只由 agent memory 内部处理）

ExtractContext

辅助对象，封装了消息范围解析（如 "0-3,7,15-20" 格式）、时间提取（年月日、星期几）、事件内容比例判断（原始消息 vs 摘要的选择）等功能，供 YAML 模板中的 Jinja2 表达式调用。

07隔离机制（Memory Isolation）

多用户、多 Agent 场景下的记忆隔离

MemoryIsolationHandler 处理多用户/多 Agent 场景：

  • 从对话消息中提取参与者的 user_id 和 agent_id
  • 根据 namespace_policy 计算记忆文件的写入目录
  • 支持 role_id 级别的隔离开关（通过配置控制）
  • 当消息中有多个 user 或 agent 时，自动在 schema 中注入 user_id/agent_id 字段

这确保了：不同用户或 Agent 的记忆不会混淆，即使是同一场对话中的多个参与者也各有各的记忆空间。

08MemoryUpdater（持久化执行器）

LLM 操作的最终执行者

MemoryUpdater 的 apply_operations() 方法是整个流程的"落地"环节：

# 执行流程 1. 遍历所有 upsert 操作，读取旧文件内容 2. 对每个字段应用对应的 MergeOp（patch/replace/sum/immutable） 3. 合并 links 和 backlinks 4. 通过 MemoryFileUtils 将数据渲染为 markdown 文件 5. 写入 VikingFS 文件系统 6. 将文件向量化（enqueue 到向量数据库用于语义检索） 7. 处理删除操作 8. 对有 overview_template 的目录生成 .overview.md 聚合文件
关键细节 MemoryUpdater 会先读取旧文件，然后针对每个字段用对应的 MergeOp 进行合并。例如，total_executions 字段的 merge_op 是 sum，就会将新旧值相加；而 best_for 字段的 merge_op 是 patch，就会在原内容中执行 SEARCH/REPLACE。

09工具系统（Memory Tools）

LLM 在 ReAct 循环中的"手和眼"

工具功能暴露给 LLM？
MemoryReadTool 读取单个记忆文件，支持行号偏移和行数限制 ✅
MemorySearchTool 语义搜索已有记忆，自动过滤 .abstract.md 和 .overview.md ✅
MemoryLsTool 列出目录内容（含文件大小） ❌ 仅系统内部

所有工具继承自 MemoryTool 抽象基类，通过 MEMORY_TOOLS_REGISTRY 注册。工具结果会被优化（截断过长内容、提取错误摘要）以减少 token 消耗。

Read 工具返回的结果自动注入 page_id 信息，让 LLM 知道该文件在 PageIdMap 中对应的数字 ID，用于后续的链接创建。

10完整数据流图

从触发到持久化的端到端流程

关键理解 Memory Extraction 不是每轮对话都触发。它是通过 session 消息数超过阈值（默认 50 条）触发，或通过 手动调用 POST /sessions/{id}/commit 触发。add_message 只负责追加消息到内存列表，commit 才触发归档和提取。

关于"50 条消息"的精确理解

这里的 50 条阈值检查的是 len(session.messages) > memory_window，其中 session 是 Bot 本地的 Session（vikingbot/session/manager.py）。每次调用 add_message() 往列表中 append 一条，所以计数 +1。

但要注意：50 条消息 ≠ 50 轮对话。在 _process_message() 中，每一轮对话至少产生 2 条消息（user + assistant）：

session.add_message("user", ...)       →  1 条
session.add_message("assistant", ...)  →  1 条

如果用户消息包含多个 ToolPart（工具调用结果），还会被拆分为多条独立消息。因此一个活跃的对话回合可能是 3-5 条消息。反推：50 条消息 ≈ 12-25 轮对话。

触发后，Bot session 被截断（只保留最近的 keep_count 条，默认 10），因此计数器 自动重置，开启下一个 50 条周期。

注意：此处检查的 Bot 本地 session（dataclass 的 list[dict]）与 OpenViking 服务端的 OpenVikingSession（含 Message 对象）是两个不同的对象。触发 consolidation 时，会创建一个新的 OpenViking session，将 Bot session 的消息全量写入后再调用 commit_async()。

Bot 自动触发链路（消息数超过 memory_window 阈值时）

# Bot 的消息处理循环（每收到一条新消息都会执行） _process_message(msg) │ ├── _evaluate_previous_response_outcome() ← 评估上一条回复的 outcome │ ├── 消息数 > memory_window (默认 50)？ │ │ │ ├── session.clone() → 截断原始 session │ │ │ └── _consolidate_viking_memory(clone) │ │ │ └── hook_manager.execute("message.compact") │ │ │ └── OpenVikingCompactHook │ │ │ └── VikingClient.commit() │ │ │ ├── client.create_session() │ ├── session.add_message()×N ← 逐条写入 │ └── session.commit_async() ← 触发 Phase 1 + 2 │ ├── _run_agent_loop() ← LLM 执行当前轮对话 │ ├── session.add_message("user", ...) ← 记录本次对话消息 ├── session.add_message("assistant", ...) └── sessions.save(session) ← 写入 JSONL 持久化

OpenViking 服务端内部的 Memory Extraction Pipeline

# Phase 1（同步）：消息归档 session.commit_async() ├── 拆分为 archive / retain 两部分 ├── 写入 archives/archive_NNN/ └── 写入 .abstract.md + .overview.md + .meta.json # Phase 2（异步后台）：Memory Extraction asyncio.create_task(_run_memory_extraction()) │ ├── 归档摘要生成（_run_archive_summary） │ ├── extract_long_term_memories() ← 用户级记忆 │ │ │ └── SessionExtractContextProvider │ ├── 加载 MemoryTypeRegistry（从 YAML 读取定义） │ ├── 构建对话历史的 message │ └── Pre-fetch：搜索/读取已有记忆文件 │ ├── extract_agent_memories() ← Agent 级记忆（技能、经验等） │ └── 两者通过 asyncio.gather() 并行执行 # ExtractLoop.run()（上述两个路径的内部核心） ExtractLoop.run() ① SchemaModelGenerator 动态生成 JSON Schema ② 构建 System Prompt ③ ┌──────────────────────────────────┐ │ ReAct Loop (最多 3 轮) │ │ LLM ↔ read/search ↔ 已有记忆 │ └──────────────────────────────────┘ ④ LLM 输出 StructuredMemoryOperations JSON ⑤ PageIdMap 将 page_id 解析为 URI ⑥ 生成 ResolvedOperations │ ▼ MemoryUpdater.apply_operations() ├── 读取旧文件内容 ├── 按 MergeOp 合并字段（patch/replace/sum/immutable） ├── 合并 links / backlinks ├── 写入 VikingFS 文件系统 ├── 向量化到 VikingDB └── 生成 .overview.md 聚合文件

自动触发链路 vs add_message/commit：为什么不会冲突？

三层防护设计确保了自动 consolidation 与常规 add_message/commit 流程之间不存在冲突：

防护 1：Clone + Truncate 模式

consolidation 运行在 session_clone（克隆）上，而原始 session 在 consolidation 开始前已被截断。后续的 add_message 写入的是截断后的原始 session，两者操作的是完全不同的 Python 对象。

session_clone = session.clone()                     # ← 克隆
session.messages = session.messages[-keep_count:]    # ← 截断原始
await self._safe_consolidate_memory(session_clone)   # ← 在克隆上运行
# ↓ consolidation 完成后
session.add_message("user", ...)                     # ← 写入截断后的原始

防护 2：顺序执行（await 阻塞）

_safe_consolidate_memory 是 await 的（不是 asyncio.create_task），所以整个流程是严格顺序的：截断 → consolidation(等待完成) → agent loop → add_message。不存在并发写入。

防护 3：每次 consolidation 创建全新的 OpenViking session

VikingClient.commit() 内部每次都先调用 client.create_session() 创建一个全新的 OpenViking session，然后往这个新 session 中 add_message + commit_async，不会操作已有的任何 OpenViking session。

唯一的注意事项：如果外部代码在 VikingClient.commit() 返回后又单独往同一个 OV session 追加消息，那些消息需要再次 commit 才能归档。但 Bot 的正常 consolidation 流程中不会出现这种情况。

Session 截断导致经验分段：冲突如何处理？

当一个完整的 session 因超过 memory_window 阈值而被截断为多个片段后，每个片段独立触发 consolidation，各自创建 OpenViking session 并提取 agent experience。这些经验之间可能产生冲突。系统通过三层自动消解机制处理：

机制 1：Same-name 合并

如果多个片段提取出相同的 experience_name，后续片段的 LLM 会读取已有经验 + 新轨迹，通过 merge_op: replace 全量重写内容。每个后续片段都是在已有经验基础上做"在线学习"式的增量优化，而非简单覆盖。

代码链路：experiences.yaml filename_template: "{{ experience_name }}.md"（同名=同URI）→ prefetch() 读取已有经验注入上下文 → LLM 按指令"Same name → updates in place"输出新内容 → resolve_operations 通过 page_id_map 解析为编辑操作 → MemoryUpdater 以 merge_op: replace 全量写入。合并发生在 LLM 推理层，系统只做"路由+写入"。

机制 2：不同名 → 共存

如果各片段提取出不同的 experience_name，它们作为独立的经验文件共存于 viking://agent/{agent}/memories/experiences/，互不干扰。后续检索时由语义搜索决定哪些经验被激活。

机制 3：supersedes 淘汰

如果新片段发现一个更优、更泛化的经验，可以设置 supersedes: "旧经验名"。系统会自动删除旧经验文件，并将其关联的所有历史轨迹链接继承到新经验，实现经验的演化升级。

真正的局限性：全局最优性的损失

这不是"冲突"，而是在线学习 vs 批处理的根本矛盾。每个片段的 LLM 只能看到自己的片段 + 已有经验，看不到后续片段的内容：

完整会话："查天气 → 订机票 → 订酒店"（连续的 3 步操作）

片段 1: LLM 提取经验 A「查询天气」   ← 不知道后面还有订票
片段 2: LLM 合并经验 A → 经验 B「订机票」  ← 看不到后面的酒店
片段 3: LLM 合并经验 B → 经验 C「订酒店」  ← 不知道前面的天气前提

最终得到 3 条独立经验，而不是 1 条"旅行行程编排"的综合经验。这在大多数场景下是可接受的——因为 agent 经验本身就是按单一用户意图拆分设计的。但如果步骤间有强上下文依赖（如订机票价格依赖天气结果），后序经验可能存在偏差。

一句话定性：冲突有消解机制兜底，但全局最优性有损失。这是为了实时性牺牲了完备性——如果一定要等整个 session 结束再一次性提取最完整的经验，记忆就得等到对话完全结束才能生效。当前设计允许 agent 在对话进行中就持续学习和进化。

11设计亮点总结

理解这个架构为什么如此设计

① YAML 驱动的声明式配置

新增记忆类型只需添加一个 YAML 文件，无需修改代码。字段类型、合并策略、文件模板全部声明式定义。

② SEARCH/REPLACE Patch

借鉴代码编辑工具思想，让 LLM 对已有记忆进行外科手术式的局部修改，而非重写整个文件，节省 token。

③ PageIdMap 间接引用

LLM 使用简单数字 page_id 引用记忆页面，系统负责映射到真实文件 URI，屏蔽路径复杂性。

④ 多层容错

FaultTolerantBaseModel + parse_json_with_stability + patch 验证失败自动重试，层层兜底。

⑤ ReAct 模式

LLM 先"探索"（read/search 已有记忆）再"决策"（输出操作），而非盲目生成，避免重复和冲突。

⑥ 多角色隔离

通过 MemoryIsolationHandler 实现 user/agent 级别的记忆隔离，不同用户记忆互不混淆。

⑦ 向量化检索

写入后的记忆自动向量化到 VikingDB，后续对话可通过语义搜索找到相关记忆。

⑧ 动态模型生成

SchemaModelGenerator 在运行时根据 YAML 动态生成 Pydantic 模型和 JSON Schema，类型安全且灵活。
